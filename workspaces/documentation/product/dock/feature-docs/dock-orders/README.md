# Dock Orders — Управление заказами на поставку

> Источник требований: REQ-DOCK-016..020, REQ-DOCK-024..025 | Слайды: Dock Order Listing, 2025 05 - SLOT MULTI ORDER MULTI CUSTOMER

---

## Концепция Dock Order

**Dock Order (DO)** — основной заказ на поставку, представляющий одну запись «что и когда прибудет/убудет» на склад. К каждому DO могут быть привязаны один или несколько **Sub Dock Orders (SDO)** — каждый SDO соответствует конкретному слоту.

---

## Листинг Dock Orders: Inbound / Outbound (REQ-DOCK-016)

Листинг разделён на два отдельных представления:

| Листинг | Условие отображения | Порядок по умолчанию |
|---------|---------------------|----------------------|
| **INBOUND** | У аккаунта активирован INBOUND | Показывается первым |
| **OUTBOUND** | У аккаунта активирован OUTBOUND | Показывается вторым |

Если у аккаунта оба направления — доступны обе вкладки. Смешивать INBOUND и OUTBOUND в одном листинге запрещено.

### Фильтры

| Фильтр | Тип |
|--------|-----|
| Supplier name | Dropdown |
| Customer name | Dropdown |
| Window Validity | Диапазон дат |
| Zone | Dropdown |

| DOCK ORDER status | Множественный выбор |
| SDO status | Множественный выбор |

Поддерживается **Magic Filter** — быстрый контекстный поиск по всем полям.

## Детали и действия в листинге (REQ-DOCK-018)

### Отображение SDO

| Элемент | Поведение |
|---------|-----------|
| Счётчик SDO | Просто число (не «(count)/x») |
| Слот ID | Кликабелен → открывает слот в новом окне (с проверкой прав) |
| Кнопка бронирования | Прямо из карточки DO → начать процесс слотирования |

### Действия по статусам

**При клике на «Missing SLOT»:**
- CLOSE ORDER — закрыть заказ
- DELETE ORDER — удалить заказ
- Modify window — изменить временной диапазон

**При клике на статус DO:**
- Если DO без SDO: CLOSE или DELETE
- Если DO закрыт без SDO или с SDO: OPEN (переоткрыть)
- При переоткрытии DO с SDO: предложить продлить окно + снять флаг MONO

## Автоматическое закрытие по сроку (REQ-DOCK-019)

Система отслеживает дедлайн доставки через поле `MAX Order Delivery Window`:

| Условие | Статус |
|---------|--------|
| MAX Window ≥ сегодня + 3 дня | **Deadline approaching** |
| MAX Window ≥ сегодня | **Deadline exceeded** → автозакрытие DO |

**Фильтр по сроку:**
- Deadline on track
- Deadline approaching
- Deadline exceeded

## Мульти-заказ / Мульти-клиент (REQ-DOCK-020)

Один слот может содержать **несколько Dock Orders** от разных партнёров.

### Условия для объединения в слот

| Условие | Требование |
|---------|-----------|
| Direction | Одинаковый (INBOUND или OUTBOUND) |
| Zone ID | Одинаковый для всех заказов в слоте |
| Customer/Supplier ID | Одинаковый |
| Смешивание IN+OUT | ❌ Запрещено |

### Ограничения

- **Линейные метры:** суммируются отдельно по каждому клиенту
- **Зона:** если у клиентов нет общей зоны — добавление невозможно
- **Zone ID может быть необязательным**, если зарегистрирован `customer_id` (INBOUND) или `supplier_id` (OUTBOUND)

## CSV и API структура Dock Order V2 (REQ-DOCK-024)

### Правила обязательных полей по типу дока

| Тип | Customer ID | Supplier ID | Booker ID | Zone |
|-----|-------------|-------------|-----------|------|
| INBOUND CUSTOMER DOCK | **Обязателен** | Опционален | Опционален | — |
| OUTBOUND SUPPLIER DOCK | Опционален | **Обязателен** | Опционален | — |
| INBOUND без CUSTOMER DOCK | Опционален | Опционален | Опционален | **Обязательна** |

### Миграция API

- CSV V2 определён в BO для загрузки пользователями
- Старый API отключается после периода миграции
- API V2 дублирует логику CSV
- Для legacy аккаунтов: вместо Customer ID принимается PARTNER ID

## Статусы Dock Order: механика и акторы (сверено с кодом 2026-06-12)

> Источник: `models/order.js:268`, `services/dock-orders/query.js:21-49`, `legacy.js`

**Зачем статусы:** заказ (dock order) — обещание поставки; статус отвечает складу на главный вопрос «придёт ли это вовремя и есть ли под это слот».

| Статус | Что означает | Кто/что переводит | Почему так |
|--------|--------------|-------------------|-----------|
| **PENDING** | Заказ создан, слота нет | Создание (CSV/API/вручную) | Стартовое состояние: обещание есть, расписания нет |
| **ASSIGNED** | Хотя бы одна строка заказа привязана к слоту | Поставщик/перевозчик бронирует слот (Slotify/SlotBook), оператор вручную — `assignOrderProductsBySlotContents()` | Заказ «встал в расписание» — склад может планировать персонал |
| **DELIVERED** | Все строки выполнены | Система при завершении всех строк (`updateOrdersStatus()`) | Факт исполнения — закрывает обещание |
| **MISSING_SLOT** | Дедлайн прошёл, слота так и нет | **Автоматически**: `NOW() > shipping_date_to+time` (OUT) / `arrival_date_to+time` (IN) — `partQueryOrderMissingSlotStatus()` | Ранний сигнал срыва: оператор видит «горящие» заказы и эскалирует поставщику ДО того, как клиент заметит |
| **CANCELLED** | Заказ отменён | Оператор (`processDeleteDockOrder` — только если все строки PENDING) | Нельзя отменить то, что уже едет |

**На что влиять, чтобы изменить статус:** PENDING→ASSIGNED — забронируйте слот под строки заказа; MISSING_SLOT уходит при назначении слота (или сдвиге окна дат заказа `PATCH /dock-orders/:id/dates`).

## Multi-client (один склад — много клиентов)

Поля `buyer_partner_id`/`seller_partner_id` на заказе и на связке слот-заказ (`slots_orders`) — склад различает, ЧЕЙ это заказ. Листинг `/dock-orders` фильтрует по партнёру (buyer для inbound, seller для outbound), зоне, статусу, окну дат; права: `accessDockOrdersPage` (просмотр) / `accessModifyDockOrder` (правки); `sharedOrdersAccountIds` в ACL открывает PML заказы связанных суб-аккаунтов. CSV-схема v2 (Buyer/Seller) согласована с этой моделью — см. [csv-uploads](../csv-uploads/README.md).
