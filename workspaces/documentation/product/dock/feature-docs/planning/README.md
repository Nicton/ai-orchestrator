# Planning — Страница планирования склада

Planning — основной интерфейс оператора склада для управления слотами. Показывает все отправки, прибывающие или убывающие из Master Location, в различных представлениях.

## Представления Planning

| Вкладка | URL | Описание |
|---------|-----|----------|
| Board | `/slotify/board` | Карточный вид (Kanban) слотов |
| Day | `/slotify/load-day` | Конкретный день с временными слотами |
| Week | `/slotify/week` | Полная неделя с процентом загрузки |
| Door Assignment | `/slotify/load/YYYY-MM-DD` | Назначение ворот (drag-and-drop) |

## Board (Доска)

Kanban-вид: слоты разбиты по колонкам в зависимости от статуса.

| Элемент | Описание |
|---------|----------|
| Карточка слота | Перевозчик + тип груза + статус + иконки |
| Цвет карточки | Соответствует статусу (зелёный/жёлтый/серый) |
| Клик по карточке | Открывает детали слота |

## Day View (День)

Показывает временную сетку для конкретного дня:
- Горизонталь: ворота (Dock Doors)
- Вертикаль: временные интервалы (шаг по `interval` зоны)
- Нагрузка: % заполненности каждого часа

**Действия:**
- Перетащить слот на другое время (Drag-and-drop → replan)
- Клик → открыть детали слота
- Добавить слот → новый слот в выбранное время

## Week View (Неделя)

Недельный вид с показателями нагрузки:

```
        Пн    Вт    Ср    Чт    Пт
08:00  [  ]  [A ]  [  ]  [B ]  [  ]
08:30  [  ]  [  ]  [C ]  [  ]  [  ]
09:00  [D ]  [  ]  [  ]  [  ]  [E ]
...
```

| Элемент | Описание |
|---------|----------|
| Ячейка | Слот с сокращённым названием перевозчика |
| % нагрузки | 100% = слот полностью заполнен (новые бронирования невозможны) |
| Цвет строки | Индикатор загруженности (зелёный → жёлтый → красный) |

## Фильтры

| Фильтр | Описание |
|--------|----------|
| Zones | Показать только выбранные зоны |
| Carriers | Фильтр по перевозчику |
| Locations | Фильтр по локации (для мульти-PML) |
| Modes | Тип транспорта |
| Reception toggle | Показать/скрыть Reception-слоты |
| Expedition toggle | Показать/скрыть Expedition-слоты |

**Эндпоинт для Planning:** `GET /api/v1/slots/planning-slots` (requireShipper)  
**Day-load view:** `GET /api/v1/slots/load-day` (requireShipper)

## Действия оператора из Planning

| Действие | Кнопка / Меню | API |
|----------|---------------|-----|
| Обновить статус | Статусный модал | `POST /api/v1/slots/:id/status` |
| Переназначить ворота | Drag & drop / выпадающий список | `PATCH /api/v1/slots/:id` |
| Перенести слот | Drag & drop на новое время | `PATCH /api/v1/slots/:id/replan` |
| Отменить слот | Меню → Отменить | `PATCH /api/v1/slots/:id/cancel` |
| Добавить комментарий | Меню → Комментарий | `PATCH /api/v1/slots/:id/comment` |
| Экспортировать день | Кнопка Excel | `GET /api/v1/slots/export-excel` |

## Dock Status Update Modal — Единый модал статусов

Заменил несколько отдельных модалов единым интерфейсом.

**Общее поведение:**
- Предзаполняет текущее время
- Поле Dock Door показывается только если не было введено ранее
- Порог on-time/late настраивается в конфигурации

**Условная логика по статусам:**

| Статус | Особое поведение |
|--------|------------------|
| `driver_called` | Проверка данных водителя: 3 кнопки (Set later / Set anyway / Provide driver) |
| `loaded` / `unloaded` | Сначала показывается модал обновления packing list; при 2+ заказах — шаг reconciliation |

**Backdating (ручной режим):**  
Если `manual_mode=true + on_time=true + closed=status`:
→ Единый модал для **всех** статусов сразу (bulk date/time update)

## Validation Slots (Слоты на валидации)

Если зона включила `is_slot_validation=true`, новые бронирования получают статус `pending_validation` и попадают на отдельную страницу для подтверждения.

| Эндпоинт | Описание |
|---------|----------|
| `GET /api/v1/validation-slots` | Список слотов на валидации |
| `GET /api/v1/validation-slots/count` | Счётчик (для бейджа в навигации) |
| `PUT /api/v1/validation-slots/:id/:status` | Approve или Decline |

**Статус после одобрения:** `new`  
**Статус после отклонения:** `declined`

## ML Status Activations в Planning

На странице Planning оператор может активировать кастомные статусы ML. При активации:
- Создаётся соответствующий TrackingPoint на Shipment
- Shipment.status обновляется через `calculateShipmentStatus()`

Настройка: Settings → Master Location → Status Activations.

## Виды Planning: что для чего (сверено с кодом 2026-06-12)

| Вид | URL | Что показывает | Когда использовать |
|-----|-----|----------------|--------------------|
| **Week** | `/slotify/:day` | Таблица зоны × дни, карточки слотов (имя, длительность, объём, статус цветом, партнёр) | Тактика недели: баланс между днями |
| **Board** | `/slotify/board/:day` | Kanban на 7 дней вперёд, карточки по зонам, отдельно Reception/Expedition | Оперативный разбор: что сегодня-завтра, клик → правка дат |
| **Load (day/week)** | `/slotify/load/:day` | Тепловая карта занятости зона × час | Узкие места и свободные мощности — [load-view](../load-view/README.md) |
| **Assignment** | там же | Сетка время × ворота, drag-and-drop назначение двери | Распределение машин по воротам в день X |

Цвета карточек: зелёный — забронировано/в норме, красный — просрочено, синий — ожидает (pending). Фильтры (режим, перевозчик, зоны, направление, клиент) сохраняются в localStorage — оператор не настраивает вид заново каждый день.

## Видимость опозданий и отмен (сверено с кодом 2026-06-12)

**Зачем:** работник склада должен с одного взгляда на доску видеть проблемные машины — не вчитываясь в даты.

| Индикатор | Условие (код) | Смысл |
|-----------|---------------|-------|
| 🟡 `pending_late` | Статус NEW/PENDING_VALIDATION/APPROVED и `!isOnTime` (now > плановое время) | Машина **опаздывает**, ещё не приехала |
| 🔴 `closed_delay` | Статус из группы CLOSED и `!isOnTime` | Обработана **с задержкой** — попадёт в статистику |
| Отмена | Статус CANCELED — карточка в цвете аномалии | Бронь снята, окно свободно |

Расчёт: `isOnTime = slotDate.isBefore(slotPlannedDate)` (`helper-planning-slots.js:154-246`); цвет — `getColorStatus(status)`; те же правила питают [TV Display](../tv-display/README.md) (табло Delayed). Кто влияет: фактические времена ставят tracking points (оператор/Peripass) — корректные отметки = честная картина опозданий.
