# PML Settings — настройки Public Master Location

> Сверено с кодом 2026-06-12 | `location_settings.js`, `location_zone.js`, `location_zone_default_setting.js`, `location_zone_date_setting.js`, `zone-constraints-settings.js`

## Зачем (бизнес-контекст)

PML (Public Master Location) — склад, который **открывает бронирование внешнему миру** (поставщикам и перевозчикам без аккаунтов). Чтобы это не превратилось в хаос, у владельца склада есть три уровня контроля: правила площадки → правила зоны → исключения на конкретные даты. Логика уровней: то, что верно для всей площадки (кто вообще может бронировать), не должно дублироваться в каждой зоне; то, что зависит от физики зоны (ворота, персонал, тип груза), живёт на зоне; разовые отклонения (инвентаризация, праздник, короткий день) не должны ломать базовое расписание.

## Уровень 1 — площадка (`location_settings`)

| Настройка | Что контролирует | Бизнес-причина |
|-----------|------------------|----------------|
| `has_slotify_access` | Включён ли публичный портал вообще | Не каждый склад готов к самообслуживанию |
| `is_supplier_control_to_book` (+текст) | Бронировать может только поставщик (не перевозчик) | Когда договорная ответственность за приёмку — на поставщике; перевозчику показывается настраиваемый текст-инструкция |
| `carrier_update_cargo_block` {inbound, outbound} | Запрет перевозчику менять груз в слоте | Защита плана приёмки: изменения груза должны идти от владельца заказа |
| `slot_naming` | Шаблон имён слотов (вкл. Same as PML) | Единый стандарт имён для всех зон — диспетчер читает календарь без расшифровки |
| `is_display_truck_information` | Показ данных ТС | Требование служб безопасности площадки |

## Уровень 2 — зона (`location_zone`)

Все ограничители зоны:

| Поле | Назначение | Кто страдает, если настроено неверно |
|------|-----------|--------------------------------------|
| `available.from / to` | Часы окна бронирования | Машины приедут в нерабочее время |
| `available.leadtime`, `settings.pre_advice` (дефолт 7 дн.) | За сколько минимум до слота можно бронировать | Склад не успеет спланировать персонал |
| `available.cutoff` | Дедлайн — позже нельзя бронировать | Поздние брони ломают план дня |
| `capacity` {duration, time_per_slot, rounded_to, interval} | Расчёт длительности и сетка окон | См. [slots-core](../slots-core/README.md) |
| `cargo_groups`, `cargo[]` | Принимаемые типы груза + производительность/час | Зона получит груз, который не умеет обрабатывать |
| `allow_outside_range` | Разрешить бронь вне окна | Точечные исключения без правки расписания |
| `allow_partial_deliveries` | Частичные поставки | Жёсткость приёмки под процесс клиента |
| `allow_add_cargo_type` | Может ли бронирующий добавить новый тип груза | Контроль над номенклатурой |
| `is_slot_validation` (+whitelist) | Ручная валидация чужих броней | См. [slot-validation](../slot-validation/README.md) |
| `priority` {order, count} | Очерёдность предложения зон и лимит слотов в приоритете | Управление, какую зону Slotify предлагает первой |
| Реплан: `replan_from/to/cutoff` | Окно, когда бронирующий может перенести слот | Баланс гибкости перевозчика и стабильности плана |
| Отмена: `cancel_from/to/cutoff` | Окно, когда можно отменить | См. ниже «Отмена и санкции» |

## Уровень 3 — расписание и календарные исключения

- **База по дням недели**: `location_zone_default_settings` — на каждый день недели свои часы loading/unloading. Почему раздельно: приёмка и отгрузка часто работают разными сменами.
- **Исключение на дату**: `location_zone_date_settings` — override часов (и rate) на конкретный день. Это и есть «поменять часы работы в календаре»: базовое расписание не трогается, исключение живёт отдельно и автоматически выигрывает у базы в этот день.

## Реплан слота

- Endpoint/логика: `replanSlot` → `replanSlotInTransaction` (`services/slots/index.js:2236-2316`)
- **Старый слот не удаляется** — меняются его даты, создаётся tracking point о переносе и пост в чат (`replanSlotPostMessage`): полная история «когда и кем перенесено» сохраняется для разбора споров.
- Кто может: бронирующий через Slotify (в пределах `replan_from/to`, не позже `replan_cutoff`) и оператор (без ограничений окна).
- Для recurring-серий можно перенести/удалить «этот и все следующие» (`removeAllNextOnesRecurringSlots`).

## Отмена и санкции

- Окно отмены: `cancel_cutoff` — позже этого порога бронирующий отменить не может (только оператор). Причина: поздняя отмена уже не позволяет продать окно другому.
- Процесс (`cancelSlot`, create_update_slot.js:768): статус → CANCELED, пустой визит удаляется, отправки получают событие BOOKING_CANCELED, пост в чат, tracking points по каждому shipment.
- **Денежных штрафов в системе нет** — есть флаг **`is_last_minute_cancellation`** (`slots.js:161`), который помечает позднюю отмену для статистики («Late cancellation %» в отчётах). Санкции — договорная практика вне системы; если потребуется автоматизация штрафов, это новая разработка.
