# Emailing — почтовые уведомления DOCK

> Сверено с кодом 2026-06-12 | `services/emails-impl.js`, микросервис `msg-email` (EJS-шаблоны `_layout/template-slots.ejs`, partials/slot/*)

## Зачем

Внешний бронирующий не сидит в системе — email это его «интерфейс статуса»: подтверждение, изменение, отмена, no-show. Без писем самообслуживание Slotify не работало бы: люди звонили бы «а прошла ли бронь?».

## Как устроено

Событие на слоте/визите → backend собирает payload (`buildSlotNotificationEmail`: адреса, дата слота, содержимое, заказы) → очередь `send_email` → микросервис **msg-email** рендерит EJS-шаблон (layout `template-slots` + partials: header / contents / modifications / orders) на языке получателя и отправляет.

## События → письма бронирующему

| Событие | Письмо |
|---------|--------|
| Бронь создана/подтверждена | Подтверждение со слот-датой и составом |
| Изменение дат/состава | «Modifications» — diff в partials/modifications |
| Отмена | Уведомление об отмене (кто/когда) |
| No-Show | Фиксация неявки |
| Смена статуса визита | Статусные нотификации |

Шаблоны едины для всех площадок (бренд — через Galaxy-логотип); язык — из языка получателя/брони. Кто влияет: оператор триггерит событиями статусов; настройки получателей — контакт компании (см. External Partners) + email брони.

## Где найти и как настроить (UI)

Получатели и события писем настройками в UI **не управляются напрямую** — письма триггерятся событиями слота/визита автоматически; адресат = email брони + контакт компании партнёра (см. [External Partners](../external-partners/README.md)). Бренд в письме — Galaxy-логотип (определяется доменом email). Тексты/языки — Tolgee (`emailing.json`).

## Сценарии

1. **Подтверждение брони**: поставщик забронировал слот → авто-письмо со слот-датой и составом на языке получателя.
2. **Отмена складом**: оператор отменил слот → письмо «бронь отменена» уходит контакту компании.
