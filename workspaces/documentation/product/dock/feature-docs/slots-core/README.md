# Slots Core — слот как бизнес-сущность (общее для TMS и DOCK)

> Сверено с кодом 2026-06-12 | Источники: `models/slots.js`, `services/slotify/duration.js`, `services/slots/`, `site_shipment_status_setting.js`

## Зачем существует слот (бизнес-контекст)

**Проблема:** склад имеет ограниченную пропускную способность (ворота, персонал, часы работы), а прибытие грузовиков без расписания создаёт очереди, простои водителей (за которые перевозчики выставляют штрафы) и пиковые перегрузки персонала.

**Решение:** слот — это **временное окно на конкретной зоне склада**, забронированное под конкретную операцию (приёмка/отгрузка). Слот «сшивает» три мира:
- **TMS-мир** (перевозка): слот привязывает Shipment к моменту погрузки/выгрузки;
- **DOCK-мир** (склад): слот занимает вместимость зоны и ворота;
- **Внешний мир** (поставщик/перевозчик без аккаунта): через Slotify бронирует слот по ссылке.

**Откуда появляется слот:** бронирование в Slotify (внешний пользователь), кнопка +BOOK в приложении (перевозчик/шиппер), оператор вручную в Planning, CSV-импорт, recurring-шаблон, автоматически при подтверждении отправки с привязкой к PML.

## Длительность слота: как считается и кто влияет

Алгоритм (`services/slotify/duration.js:45-117`) — приоритет сверху вниз:

| # | Правило | Когда применяется | Кто влияет |
|---|---------|-------------------|-----------|
| 1 | **Факт**: `closed_TP − ongoing_TP` | Слот уже отработан (есть tracking points начала и конца) | Оператор, фиксирующий статусы |
| 2 | **Ручная**: `slot.duration` | Оператор задал длительность явно | Оператор |
| 3 | **По грузу**: `Σ(qty/per_hour + stackedQty/stacked_per_hour)×60 + time_per_slot` | В зоне настроены cargo-типы | Настройки зоны (PML-админ) + объём груза (бронирующий) |
| 4 | **Дефолт зоны**: `zone.capacity.duration` | Нет cargo-настроек | PML-админ |
| 5 | **Округление**: вверх до кратного `rounded_to` | Всегда в конце | PML-админ |

**Почему так:** склад планирует персонал по производительности (паллет/час), поэтому время честно выводится из объёма; `time_per_slot` — фиксированные накладные расходы на каждую машину (постановка к воротам, документы); округление — чтобы календарь оставался читаемым (кратно 15/30 мин).

**Пример:** зона: time_per_slot=10, rounded_to=15; груз: 30 палет @50/час + 20 stacked @100/час → (30/50 + 20/100)×60 = 48 мин → +10 = 58 → округление → **60 минут**.

## Вместимость (capacity) и доступные окна

- `zone.capacity.interval` (дефолт 30 мин) — шаг сетки календаря: бронируемые окна 08:00-08:30, 08:30-09:00…
- Доступность окна = в нём помещается рассчитанная длительность и нет конфликтующих слотов.
- `allow_partial_deliveries` — разрешить частичную приёмку; `allow_outside_range` — бронирование вне основного окна зоны.
- Жёсткого «N слотов в день» нет — лимит возникает естественно из (часы работы × ворота ÷ длительность). Прогнозную загрузку оператор видит на [Load-странице](../load-view/README.md).

## Типы груза зоны

`zone.cargo_groups` — какие типы груза зона принимает вообще (валидация при бронировании); `zone.cargo[]` — производительность по каждому типу (`per_hour`, `stacked_per_hour`, опционально per-shipper). `allow_add_cargo_type=false` запрещает бронирующему добавлять не настроенные типы. Лимит «хранения в час» — это и есть per_hour: больше груза в час, чем настроено, зона физически не обработает — длительность слота растянется.

## Статусы слота: полный жизненный цикл с обоснованием

17 внутренних статусов (`models/slots.js:237`) сгруппированы по смыслу (`site_shipment_status_setting.js`):

```
NEW ──(валидация зоны?)── PENDING_VALIDATION ──► APPROVED / DECLINED
 │
 ▼ (день визита)
DRIVER_CALLED → ON_SITE → AT_DOCK → LOADING/UNLOADING → LOADED/UNLOADED → LEFT_DOCK → LEFT_SITE
 │
 └──► CANCELED / REFUSED / NO_SHOW  (аномалии)
```

| Группа | Статусы | Зачем нужна (бизнес-смысл) |
|--------|---------|---------------------------|
| Планирование | NEW, APPROVED, PENDING_VALIDATION, DECLINED | Бронь существует, ресурс зарезервирован; валидация — фильтр «чужих» перевозчиков (см. [slot-validation](../slot-validation/README.md)) |
| ONGOING | DRIVER_CALLED…LOADING, ADMIN_CLEARED_* | Машина в процессе — по этим статусам склад видит, что происходит на площадке прямо сейчас |
| CLOSED | LEFT_DOCK, и далее | Операция завершена, ресурс освобождён, факт зафиксирован для KPI |
| ANOMALY | REFUSED, NO_SHOW, CANCELED | Исключения для статистики надёжности поставщиков/перевозчиков |

**Кто переводит** (матрица акторов):

| Переход | Актор | Механизм |
|---------|-------|---------|
| NEW → PENDING_VALIDATION | Система | Зона с `is_slot_validation=true`, бронирующий не в whitelist |
| PENDING_VALIDATION → APPROVED/DECLINED | Оператор склада | Очередь валидации |
| → DRIVER_CALLED / ON_SITE / AT_DOCK | Оператор (модал статуса) или **Peripass-гейт автоматически** | Tracking point на слоте |
| → LOADING/UNLOADING → LOADED/UNLOADED | Оператор дока | Модал + packing-list reconciliation |
| → CANCELED | Бронирующий (в пределах cancel-окна зоны) или оператор | `cancelSlot` |
| → NO_SHOW / REFUSED | Только оператор | Машина не приехала / отказ на месте |

**Почему статусы LOADED и UNLOADED разные:** направление операции (Expedition vs Reception) — для склада это разные процессы и разные KPI; публичный статус наружу схлопывается в Delivered/Loaded.

Каждому статусу соответствует tracking point с временной меткой (`slot_tracking_points`) — из них считается фактическая длительность (правило №1 выше) и вся статистика (опоздания, no-show rate, среднее время обработки).

## Связь со слотами TMS-отправок

Слот ссылается на shipment-контент (`slot_contents`), визит (`visit_id`) и заказы (`slots_orders`). Подтверждение слота двигает статус отправки (SLOT_CONFIRMED на Shipment) — так перевозка и склад остаются синхронными без ручной сверки.

## Уточнения по capacity (по вопросам Product, 2026-06-12)

- **«Сколько слотов в день на зоне»** — прямого поля-лимита нет; ёмкость = часы работы × двери ÷ длительность. Точечные лимиты есть на **recurring-слотах**: `max_linear_meter` (метраж в слот), `default_workload`, `range_per_slot` (машин в слот); заполняемость считается как `fillRate = забронированный метраж / max_linear_meter` — >100% красный, право `CAN_FORCE_SLOT` позволяет пробить (оранжевый), `lockMin/MaxTime` — жёсткий замок, который не пробивается.
- **«Максимальный объём хранения»** на зоне — поля нет (НЕ РЕАЛИЗОВАНО); управляйте через per_hour производительность и метраж recurring-слотов. Если нужен storage-лимит — feature request.
