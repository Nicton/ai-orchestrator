## Два процесса

Система всегда запускается как **два отдельных Node.js процесса**:

| Процесс | Файл | Порт | Назначение |
|---------|------|------|-----------|
| API сервер | `app/server.js` | 3013 | HTTP запросы от клиентов |
| Worker | `worker/index.js` | 3018 | Фоновые задачи из Redis-очереди |

Оба подключаются к одной БД и одному Redis. Событийные обработчики интеграций инициализируются в обоих процессах.

## Поток обработки HTTP-запроса

```
Входящий запрос
    │
    ▼ [1] Middleware
    │  • Логирование запроса
    │  • Парсинг body/cookies
    │  • Извлечение JWT токена
    │  • Загрузка User из БД
    │  • Загрузка Account пользователя
    │  • Проверка ролей и разрешений
    │  • Загрузка ACL-контекста (divisions, locations)
    │
    ▼ [2] Route Handler
    │  • Матчинг паттерна маршрута
    │  • Route-специфичный middleware
    │
    ▼ [3] Controller
    │  • Парсинг параметров запроса
    │  • Валидация (Joi / кастомная)
    │  • Вызов сервисного слоя
    │
    ▼ [4] Service Layer
    │  • Бизнес-логика
    │  • Запросы к БД через модели
    │  • Постановка async-задач в очередь
    │
    ▼ [5] Model Layer
    │  • SQL запросы
    │  • Хуки модели
    │
    ▼ [6] Controller (ответ)
    │  • Формирование JSON-ответа
    │  • HTTP статус
    │
    ▼ Ответ клиенту
```

## Поток async-задачи

```
Сервисный слой
    │  • Создаёт данные задачи
    │  • Кладёт в Kue-очередь
    │  • Возвращает управление немедленно
    ▼
Redis (Kue)
    │  • Хранит задачу
    │  • Отслеживает состояние
    ▼
Worker Process
    │  • Забирает задачу
    │  • Выполняет handler
    │  • Загружает данные из БД
    │  • Вызывает внешние API
    │  • Обновляет БД
    │  • Отправляет уведомления
    ▼
Завершение / Retry при ошибке
```

## Поток интеграции с перевозчиком

```
API: Booking Shipment

    ▼

    │  • Создаёт shipment в БД
    │  • Ставит задачу интеграции в очередь
    ▼
Integration Queue
    ▼

    ▼
integration/[carrier]/service.js
    ▼

    └─▶ provider.js       — HTTP-запрос к API перевозчика

    ▼
Уведомление пользователя (email/webhook)
```

## Многоарендность (Multi-Tenant)

Каждый запрос изолирован по аккаунту:

```
Пользователь входит
    ▼
User.account → определяем тип аккаунта
    │
    ├─ SHIPPER → ShipperACL (divisions, locations)
    │            Фильтрация запросов по shipper_id, division_ids
    │
    └─ CARRIER → CarrierACL (divisions)
                 Фильтрация по carrier_id + allowed shippers
```

## Событийная система

```
Изменение статуса Shipment
    ▼
Транзакция в БД
    ▼
After-Commit хук
    │
    ├─▶ Локальный Event Bus
    │   └─▶ Listeners:
    │       • Email-уведомление в очередь
    │       • Webhook в очередь
    │       • Обновление Elasticsearch
    │       • Запуск обновлений интеграций
    │
    └─▶ Kafka Event
        └─▶ Внешние потребители (микросервисы)
```

## Безопасность (Defence in Depth)

| Слой | Механизм |
|------|---------|
| Сеть | HTTPS, CORS, Rate limiting |
| Аутентификация | JWT + отзыв токенов через Redis |
| Авторизация | RBAC + ACL (divisions, locations) |
| Приложение | Joi валидация, XSS-санитизация, Helmet |
| Данные | ORM защита от SQL-инъекций, мягкое удаление (paranoid) |
