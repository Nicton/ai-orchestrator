# Домены и сущности TMS

> Источники: Confluence TD space (01_PRODUCT_MAP, 03_OBJECTS_STATES_TRANSITIONS), doc/model-architecture.md

## Основные домены продукта

```
TMS (Transport Management System)
│
├── Booking / CSW (Create Shipment Workflow)
│   └── Входная точка создания перевозки
│
├── Requests
│   ├── SR (Shipment Request) — заявка одному перевозчику
│   └── QR (Quote Request)   — запрос котировок нескольким перевозчикам
│
├── Shipments — подтверждённые перевозки
│   └── Вкладки: Booking · Tracking · Invoicing · Claims
│
├── Tracking — отслеживание грузов
│   └── Tracking Points (TP)
│
├── Invoicing — выставление счётов
│
├── Slots — тайм-слоты на складе/воротах
│
└── Milkrun — связанные перевозки (один маршрут, несколько отправлений)
```

## Стейт-машина: от черновика до Shipment

```
Draft (черновик)
    │  CSW step1 → минимальные данные → SAVE
    │
    ├─▶ SR (Shipment Request)
    │       │ → auto-confirm или ручное подтверждение
    │       ▼
    │   Shipment ──────────────────────────────────────────┐
    │                                                      │
    └─▶ QR (Quote Request)                                 │
            │ → получение котировок от перевозчиков        │
            │ → выбор котировки                            │
            ▼                                              │
        Shipment ──────────────────────────────────────────┘
            │
            ├─ Booking вкладка  (детали бронирования)
            ├─ Tracking вкладка (статусы + Tracking Points)
            ├─ Invoicing вкладка (счета)
            └─ Claims вкладка   (претензии)
```

### Статусы Shipment Request (SR)

| Статус | Описание |
|--------|---------|
| Draft | Черновик, минимум данных |
| RTB (Ready-to-Book) | Заполнен, готов к отправке |
| Pending | Отправлен перевозчику, ожидает ответа |
| Confirmed | Перевозчик подтвердил |
| Cancelled | Отменён |

### Tracking Points (TP)

Ключевые действия по TP, доступные через hover → **Show actions**:
- **confirm** — подтвердить точку
- **replan** — перепланировать
- **request info** — запросить информацию у перевозчика

## Ключевые сущности и модели

### Account (корень multi-tenant изоляции)

```
Account
  ├─ hasMany → User
  ├─ belongsTo → Shipper  (если shipper-аккаунт)
  └─ belongsTo → Carrier  (если carrier-аккаунт)
```

Модель: `app/models/account.js`

### Shipper (грузоотправитель)

```
Shipper
  ├─ hasMany → ShipmentRequest
  ├─ hasMany → Shipment
  ├─ hasMany → ShipperDivision  (структурные подразделения)
  └─ hasMany → Slot
```

Модель: `app/models/shipper.js`

### Carrier (перевозчик)

```
Carrier
  ├─ hasMany → Slot
  ├─ hasMany → CarrierDivision
  └─ belongsToMany → Shipper  (через партнёрства)
```

Модель: `app/models/carrier.js`

### ShipmentRequest (заявка на перевозку)

```
ShipmentRequest
  ├─ belongsTo → Shipper
  ├─ hasOne  → QuoteRequest  (если QR)
  └─ hasMany → Shipment      (подтверждённые перевозки)
```

Модели: `app/models/shipment_request.js`, `app/models/quote_request.js`

### Shipment (перевозка)

```
Shipment
  ├─ belongsTo → Carrier
  ├─ belongsTo → ShipperDivision
  ├─ belongsTo → CarrierDivision
  ├─ belongsTo → ShipmentRequest
  ├─ belongsToMany → Slot  (через SlotShipment)
  └─ hasMany → TrackingPoint
```

Модель: `app/models/shipment.js`

### Slot (тайм-слот)

```
Slot
  ├─ belongsTo → Carrier
  ├─ belongsToMany → Shipment  (через SlotShipment)
  └─ hasMany → SlotShipment
```

Модель: `app/models/slot.js`

## ACL — структура доступа

```
User
  ├─ belongsTo → Account (активный аккаунт)
  ├─ hasMany → ShipperACL  (доступ к divisions грузоотправителя)
  ├─ hasMany → CarrierACL  (доступ к divisions перевозчика)
  └─ hasMany → UserLocationZone  (доступ к локациям)
```

ShipperACL задаёт: `shipper_id + division_id + permissions`
CarrierACL задаёт: `carrier_id + division_id + permissions`

## Milkrun

Два варианта создания:
- **CSW Milkrun**: один booking → несколько shipments через несколько pre-shipments
- **Grouping Milkrun**: кнопка `[MILKRUN]` на grouping page → Create Milkrun Modal

Условие Milkrun (Qase 1625): во втором pre-shipment должны совпадать **pick-up или delivery locations и дата** с предыдущим.
