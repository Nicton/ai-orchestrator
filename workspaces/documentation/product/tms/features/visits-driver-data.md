# Visits — Данные водителя и грузовика

> Источник требований: REQ-VIS-001..007 | Слайды: 2024 01 - VISITS drivers

---

## Концепция

Функция разделяет данные о водителе и грузовике на два независимых набора: **Expected** (ожидаемые, из системы) и **Checked** (проверенные, введённые оператором на месте).

**Цель:** зафиксировать расхождения между тем, кто ожидался, и тем, кто фактически прибыл.

---

## Разделение Expected / Checked (REQ-VIS-001)

| Тип данных | Источник | Кто обновляет |
|-----------|---------|--------------|
| **Expected Driver data** | API / Slotify / SlotBook / бронирование | Система автоматически |
| **Checked Driver data** | Оператор склада на месте | Только вручную через UI |
| **Expected Truck data** | Аналогично Driver | Система |
| **Checked Truck data** | Оператор | Только вручную |

Все источники (API, Slotify, SlotBook) обновляют **только EXPECTED** данные.

---

## UI блока водителя в карточке визита (REQ-VIS-002)

- Блок оформлен светло-серым фоном для читаемости
- Номер документа водителя: последние 3 цифры заменены на `***`
- Кнопка SMS — только если указан номер телефона
- При отсутствии данных — список недостающих полей (Missing data)
- Поле TRUCK: максимальная ширина под 10 символов

---

## Идентификация заблокированных водителей (REQ-VIS-003)

| Этап | Визуальный индикатор |
|------|---------------------|
| При идентификации заблокированного (Expected) | Светло-оранжевый блок |
| При заполнении Checked данных с заблокированным | Тёмно-оранжевый / красный блок |

- Пользователь видит предупреждение с номером (#XX) заблокированного водителя
- Возможность зарегистрировать заблокированного водителя, а затем отказать в доступе на уровне VISIT
- Случай заблокированного водителя = **критический инцидент**

---

## Процесс заполнения CHECKED данных (REQ-VIS-004)

| Роль | Поведение |
|------|----------|
| **ADMIN** | Видит модал обновления EXPECTED DRIVER → затем модал CHECKED DRIVER |
| **Не-admin** | Пропускает шаг обновления EXPECTED (security) → сразу к CHECKED |

Модал CHECKED DRIVER предзаполняется из EXPECTED DRIVER.

Поля документа предзаполняются с **тремя последними символами** скрытыми (пользователь вводит полный номер для верификации).

Кнопка «Save in Database» доступна **только** в разделе CHECKED DRIVER.

---

## Валидация CHECKED данных (REQ-VIS-005)

| Ситуация | Индикатор |
|----------|----------|
| Введено меньше символов, чем в EXPECTED | Красная рамка |
| Введено = EXPECTED | Зелёный индикатор ✅ |
| Введено ≠ EXPECTED | Красный индикатор ❌ |
| EXPECTED поле пусто | Принимается любое значение |

Для грузовика проверяются как **LICENCE PLATE**, так и **TRAILER**.

---

## Логирование расхождений (REQ-VIS-006)

| Флаг | Условие записи |
|------|---------------|
| `Driver_data_gap = 1` | Данные водителя не совпадают с ожидаемыми |
| `Truck_data_gap = 1` | Данные грузовика не совпадают с ожидаемыми |

- Пользователь видит предупреждение перед сохранением
- После сохранения CHECKED данных → блок визита обновляется **синим фоном** (verified)
- Флаги gap позволяют в будущем фильтровать VISIT с расхождениями

---

## Сохранение водителя в базе (REQ-VIS-007)

Водитель идентифицируется по комбинации: **тип документа + номер документа**.

- Если водитель не существует в БД → создаётся новая запись
- После сохранения Checked Driver → синий фон (verified)
- Логика сохранения в базу применяется **только** для CHECKED данных, не для EXPECTED
