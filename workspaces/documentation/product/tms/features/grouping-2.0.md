# Grouping 2.0 — Группировка отправлений

> Источник требований: REQ-GRP-001..008 | Слайды: 2022 06 - Grouping 2.0, 2022 12 - MD Docs vs. BK_SH vs. Grouping

---

## Концепция

**Grouping** — объединение нескольких Shipment Requests (SH) под единым `Grouping ID` для совместного управления: документами, трекингом, уведомлениями и манифестом. Grouping 2.0 устраняет ограничения первой версии — больше не нужно «освобождать» отправления перед повторной группировкой.

---

## Типы группировки (REQ-GRP-005)

| Тип | Перевозчик | SH | Pick-up | Доставка |
|-----|-----------|-----|---------|---------|
| **Grouped Departure** | 1 | N | 1 | 1 |
| **Grouped Delivery** | 1 | N | 1 | N |
| **Grouped Shipments** | 1 | N | N | 1 |
| **Milkrun** | 1 | N | N | N |
| **Multi Container** | 1 | N | 1 | 1 (особый тип — отдельный документ) |
| **Mono Shipment** | 1 | 1 | 1 | 1 (не группа) |

Grouping ID генерируется при создании группы и отображается в карточке SH и в листинге.

---

## Создание группы из одного SH (REQ-GRP-001)

С Grouping 2.0 группу можно создать даже при выборе **одного** отправления:

| Кнопка | Условие показа |
|--------|---------------|
| **GROUP DEPARTURE** | При выборе 1+ SH |
| **GROUP ARRIVAL** | При выборе 1+ SH |

- Выбор 1 SH открывает тот же UX-поток создания группы, что и при выборе 2+ SH
- Опция создания манифеста доступна даже при группировке из 1 SH
- Grouping ID генерируется

---

## Группировка с уже существующей группой (REQ-GRP-002)

При выборе набора SH, где часть уже сгруппирована, открывается **модальное окно с вариантами:**

| Вариант | Действие |
|---------|----------|
| Создать новую сгруппированную отправку | Генерируется новый Grouping ID |
| **Добавить к существующей группе** | Используется текущий Grouping ID (не создаётся новый) |
| Создать новый Milkrun | Новый Milkrun ID |
| Добавить к существующему Milkrun | Текущий Milkrun ID |

После операции все задействованные SH получают актуальный Grouping ID.

> **Ключевое изменение v2.0:** предварительный шаг «освобождения» (FREE) перед повторной группировкой больше **не требуется**.

---

## Освобождение (FREE) из группы (REQ-GRP-003)

### Правила FREE

| Сценарий | Поведение |
|---------|----------|
| FREE одного SH из группы из **2** | Оставшийся SH сохраняет Grouping ID и Manifest ID |
| FREE одного SH из группы из **3+** | Оставшиеся SH сохраняют Grouping ID и Manifest ID |
| Удаление Grouping ID у последнего SH | Необходимо выбрать **оба** SH и применить FREE |

Для полного удаления Grouping ID у оставшегося одиночного SH — нужно выбрать оба и применить FREE.

История событий (audit log) фиксирует операцию FREE с указанием пользователя и временной метки.

## Отображение Grouping ID (REQ-GRP-004)

| Место | Поведение |
|-------|----------|
| Карточка SH | Grouping ID виден независимо от наличия манифеста |
| Листинг SH | Grouping ID в колонке, поддерживает фильтрацию и сортировку |
| Модальное окно группы | Grouping ID + ссылка «SEE THE N SHIPMENTS» |
| PLANNING VIEW | Модальное окно группы содержит Grouping ID + «Voir les N expéditions» |

Клик на «SEE THE N SHIPMENTS» → листинг фильтруется по Grouping ID.  
Активный фильтр отображается в шапке с кнопкой сброса (X / корзина).

## Загрузка документов в группе (REQ-GRP-006)

При загрузке документа на один SH из группы пользователю предлагается выбрать **область распространения**:

| Вариант | Условие |
|---------|---------|
| **This shipment** | Только текущий SH |
| **All shipments** | Все SH в группе |
| **Selection** | Конкретные SH из группы |
| **Same Ship** | Контейнеры на одном судне (только MC / Sea Freight) |
| **Same Destination** | SH с одинаковым пунктом доставки (только Grouped Delivery) |
| **Same Pick-up** | SH с одинаковой точкой погрузки (только Grouped Shipments) |

---

## Уровни доступа к документам (REQ-GRP-007)

| Уровень | Видимость | Особенности |
|---------|-----------|------------|
| **Private** | Только загрузивший аккаунт | — |
| **Limited** | Shipper + Carrier | — |
| **Public** | Все аккаунты с доступом к SH | Новые участники, добавленные позже, **автоматически** получают доступ |
| **Specific** | Выбранные вручную участники | Carrier, Shipper, Pick-up site, Delivery site, Spectator |

В листинге документов отображается тег (Private / Limited / Specific) и количество уведомлённых пользователей. При наведении — детальный список.

---

## Уведомления о документах в группе (REQ-GRP-008)

| Правило | Описание |
|---------|----------|
| Адресность | Уведомления отправляются только аккаунтам с доступом к документу |
| Тег-счётчик | `«12»` в листинге вместо полного списка; mouseover → список аккаунтов и уведомлённых пользователей |
| Public-документ | При удалении аккаунта из SH — документ остаётся доступным оставшимся участникам |
| Профиль пользователя | Вкладка «Notifications» → управление уведомлениями по документам |
| Carrier | Может настроить уведомления по конкретным клиентам (Customer-level filtering) в своём профиле |

## Manifest

Manifest — дополнительный объект к группе. Создаётся **опционально** при группировке.

| Поведение | Описание |
|-----------|----------|
| Grouping без Manifest | Допустимо — Grouping ID присваивается независимо от манифеста |
| Manifest при 1 SH | Доступна опция создания (нет ограничений по количеству) |
| Manifest ID | Сохраняется при FREE, аналогично Grouping ID |

## Влияние на DOCK / PML (добавлено 2026-06-12, по Product)

Группировка затрагивает складскую сторону: при объединении отправок происходит **слияние их слотов в один главный слот** группы. Следствие для PML: **старые слоты участников отменяются** — освобождаются окна в календаре зоны, визиты без оставшихся слотов удаляются, бронирующие получают уведомления об отмене. Планируя группировку, учитывайте: повторное бронирование главного слота подчиняется правилам зоны (cutoff, валидация) — группировать «в последний момент» может не получиться.
