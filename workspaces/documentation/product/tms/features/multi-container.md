# Multi Container — Мульти-контейнерные бронирования

> Источник требований: REQ-MC-001..008 | Слайды: 2020 02 - Multi Container

---

## Концепция

Multi Container (MC) — режим бронирования, при котором несколько физических контейнеров объединяются в одно бронирование. Каждый контейнер представляется как отдельная отправка (SH) со своим суффиксом, но управляется через единый экран бронирования.

**Условие активации MC:**
- Тип груза: FCL или FTL
- Количество (Qty): > 1
- При Qty = 1 мульти-контейнерный режим не активируется

**Особенности:**
- Стоимость управляется на уровне бронирования, а не отдельных контейнеров
- До подтверждения бронирования интерфейс идентичен стандартному

> ✅ ОТВЕТ ИЗ КОДА (2026-06-11): Максимум — **20 контейнеров по умолчанию**, переопределяется полем `max_containers_number` аккаунта/пользователя (`backend/app/services/preShipments/index.js:933`; валидация в `lib/contents.js:477`).
> ✅ ОТВЕТ ИЗ КОДА: Режим **не привязан к транспортному режиму** — MC включается при наличии контента с флагом `is_container` (`services/shipments/helper.js:154`), проверки FCL/FTL нет. Технически работает для любого режима с контейнерным контент-типом.

## Именование контейнеров при подтверждении (REQ-MC-002)

При подтверждении бронирования система предлагает задать имя/суффикс для каждой отправки:

- Перевозчик может указать номер контейнера в поле имени
- Каждый контейнер получает уникальный суффикс: `0001`, `0002`, … `000N`
- Суффиксы отображаются на экране трекинга рядом с именем отправки
- После подтверждения пакинг-лист **не может** быть изменён на уровне бронирования

> ✅ ОТВЕТ ИЗ КОДА: Переименовать **можно** в любой момент — `PATCH /shipments/:id/name` обновляет `container_name` при `is_multicontainer` (`services/shipments.js:4197`). Проверяется только ACL (shipper-владелец / carrier-исполнитель), статус подтверждения не блокирует. При ошибочном номере перевозчик просто исправляет имя тем же эндпоинтом.

## Экран бронирования в MC-режиме (REQ-MC-003)

После подтверждения бронирования:

| Элемент | Описание |
|---------|----------|
| Средняя колонка | Список всех отправок с суффиксами (дата отправки, дата прибытия, статус трекинга) |
| Кнопка | «See all containers activity» — сводная активность по всем контейнерам |
| Чат | Переключатель: все контейнеры / только уровень бронирования |

Переключатель чата **не меняет список подписчиков**, только фильтрует отображаемые сообщения.

## Управление метаданными (REQ-MC-004)

При добавлении метаданных в MC-отправку:

1. Система показывает выбор: «все контейнеры» или выбранные
2. Отображается «Update Metadata on all containers» с чекбоксами для каждого контейнера
3. Доступны кнопки **select all** / **unselect all**
4. Изменения применяются только к выбранным контейнерам

---

## Загрузка документов (REQ-MC-005)

При загрузке документа в MC-бронировании:

```
Диалог "Share documents on":
├── This container only
└── Multiple containers
    ├── [✓] Container 0001
    ├── [✓] Container 0002
    ├── [ ] Container 0003
    ├── Select all / Unselect all
    └── [CONFIRM]
```

Документ сохраняется **отдельно** для каждого выбранного контейнера.

Для Sea Freight MC: дополнительный вариант **«Same Ship»** — документ применяется ко всем контейнерам на том же судне. Подробнее в [sea-freight-ship-data.md](./sea-freight-ship-data.md).

## Трекинг нескольких контейнеров (REQ-MC-006)

При подтверждении доставки:

1. Система предлагает выбор: «all containers» или список конкретных
2. Пользователь может перейти к другому контейнеру из текущего экрана
3. Счётчик: «5 other containers on this booking»
4. Клик по счётчику — список всех контейнеров со статусами
5. Из списка можно перейти к любому контейнеру напрямую

> ✅ ОТВЕТ ИЗ КОДА: **Не блокирует** — статусы контейнеров-«братьев» (общий `pre_shipment_id`, сервис `shipment_brothers.js`) полностью независимы друг от друга.

## Инвойсинг MC-бронирований (REQ-MC-007)

Особенности отображения в реестре инвойсов:

- MC-бронирование отображается как **одна строка** в реестре
- При раскрытии — список отправок с их статусами
- Указывается количество забронированных контейнеров
- Для FCL/FTL: в пакинг-листе не отображаются размеры единиц
- При N контейнерах: даты группируются (не N отдельных строк дат)

> ⚠️ ПО КОДУ (2026-06-11): MC-специфичной invoicing-логики **нет** — счета формируются на уровне shipment_request без дифференциации по контейнерам и без фильтра по `canceler_id`. Поведение при частичной отмене кодом не определено → требует решения Product.

---

## Отмена отдельных контейнеров (REQ-MC-008)

- Отмена выполняется через раздел **TRACKING**
- Отмена одного контейнера **не отменяет** всё бронирование
- Статус отменённого контейнера обновляется в списке бронирования
- Инвойс корректно отражает фактическое количество доставленных контейнеров

> ⚠️ ПО КОДУ (2026-06-11): Отмена (`canceler_id`) доступна **обеим сторонам** в рамках их прав на бронирование (carrier — если назначенный `carrier_id`; shipper — владелец или право `CAN_EDIT_BOOKING_KEY`). **Расхождение с требованием выше:** в коде установка `canceler_id` вызывает `cancelShipmentRequestById()` — отменяется **всё бронирование**; механизм отмены строго одного контейнера не найден (`services/shipments.js:1969-1976`).
