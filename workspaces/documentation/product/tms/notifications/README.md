# Notifications — Уведомления

Система уведомлений информирует Shipper и Carrier о всех важных событиях в перевозке. Уведомления приходят по email и в интерфейсе (in-app).

## Кто получает уведомления

| Событие | Shipper | Carrier |
|---------|---------|---------|
| Создана заявка SR/QR | — | ✅ |
| Carrier подтвердил перевозку | ✅ | — |
| Carrier отклонил перевозку | ✅ | — |
| Получена котировка (QR) | ✅ | — |
| QR — котировка не выбрана | — | ✅ |
| Pick-up подтверждён | ✅ | — |
| Доставка подтверждена | ✅ | — |
| Attachment загружен | ✅ | ✅ |
| Локация обновлена | ✅ | ✅ |
| Груз изменён (Cargo update) | ✅ | ✅ |
| Дата изменена | ✅ | ✅ |
| Срок ответа на QR истекает | — | ✅ |
| Pre-Invoice создан | ✅ | — |
| Invoice утверждён | — | ✅ |

## Настройки уведомлений

По умолчанию уведомления включены для всех событий. Каждый пользователь может:
- Отписаться от конкретных типов уведомлений в настройках аккаунта
- Настроить email-дайджест (мгновенно / раз в час / раз в день)

**Followers (Follower in Chat):** Пользователь может подписаться на уведомления по конкретной перевозке через кнопку "Follow" — даже если он не участник этой перевозки напрямую.

## Бэкенд

- Worker task: `worker/tasks/mailStatus.js` — email уведомления о статусе
- Worker task: `worker/tasks/notify_*.js` — группа задач по типам уведомлений
- Cron задачи: уведомления о приближающихся дедлайнах ответа
- Kafka: события изменений → worker queue → email/push

## Внутренние мутации

При создании уведомления:
- `Notification` запись создаётся в БД
- Kue: задача `mailStatus` ставится в очередь
- SendGrid: email отправляется получателю

## Бизнес-лог событий (REQ-NOTIF-001..002)

Центр уведомлений расширяется новой вкладкой **Business Log** — журналом всех изменений объектов на уровне аккаунта.

**Структура записи лога:**

| Поле | Значение |
|------|----------|
| WHO | User + email |
| HOW | Manual / Import / API / Integration |
| WHAT | Объект (BOOKING, SHIPMENT, SLOT, VISIT, etc.) |
| WHY | Created / Updated / Cancelled / Deleted / Replanned / Validated / Declined |
| WHEN | Дата и время |
| Object ID | Кликабельная ссылка к объекту |
| Object Name | Название объекта |

**Объекты, которые логируются:** BOOKING, SHIPMENT, SLOT, VISIT, TPT REQUEST, Freight Unit, CLAIM, INVOICE LINE, INVOICE ORDER, TM ORDER, DOCK.

**Срок хранения:** 2 недели.

**Детали по клику:** боковой модал — user, email, phone, payload (для API-ошибок), список уведомлённых пользователей.

**Фильтры бизнес-лога:**

| Фильтр | Варианты |
|--------|----------|
| Users | Кто выполнил действие |
| States / WHY | Created, Updated, Cancelled, Deleted, Replanned, Validated, Declined, ERROR |
| Object / WHAT | BOOKING, SHIPMENT, SLOT, и т.д. |
| How | Manual, Import, API |
| Date | День, неделя, диапазон; последний час / 2 ч / 4 ч |
| Notify | Nobody / ALL / Me / выбранные получатели |

Поддерживается Magic Filter с сохранением в памяти пользователя.

## Настройки уведомлений на уровне пользователя (REQ-NOTIF-003)

Уведомления настраиваются отдельно для каждого аккаунта пользователя.

| Режим | Поведение |
|-------|----------|
| BOOKER / CREATOR | Уведомления по умолчанию включены (все события своих объектов) |
| Участник группы (team member) | Уведомления о действиях других бронировщиков |
| Уровень аккаунта (ACCOUNT) | Подписка на события всего аккаунта |
| SPECTATOR аккаунта | Только объект SHIPMENT (расширится позднее для SLOT/DOCK) |

При multi-account настройки задаются отдельно для каждого аккаунта.
Кнопка **Copy settings** — перенести настройки между аккаунтами с похожей деятельностью.

## Каналы доставки уведомлений (REQ-NOTIF-004..005)

### In-App (bell-нотификатор)

- Улучшенный нотификатор с кликом для перехода к объекту
- В multi-account режиме: красная точка отображается в ribbon при переключении аккаунтов

### Email

| Параметр | Описание |
|----------|----------|
| YES/NO | Пользователь выбирает, получать ли email |
| SPOT | Одно письмо на каждое событие (текущее поведение) |
| Digest | Консолидация событий через CRON: каждые 1 / 2 / 3 / 4 / 8 часов |
| Daily digest | С учётом таймзоны аккаунта (опционально) |

Если в период digest не было событий — письмо не отправляется.

**Формат digest-письма:**

Секции по бизнес-объектам: BOOKING, TRACKING, SLOT.

Каждая строка: `<user> (<account>) (action) (element) at <date&time> SEE HERE`

Пример: `ERWAN (ERWAN QA) updated packing list at 17/09 15:21 SEE HERE`

Разные категории могут иметь разную частоту digest. Если у пользователя несколько подписок (BOOKER + ACCOUNT) — одно письмо, без дублирования.

## Vacation Mode — делегирование уведомлений (REQ-NOTIF-006)

- Пользователь включает **Vacation Mode** → уведомления на свой адрес останавливаются
- Опционально: переадресовать уведомления коллеге-замещению
- Все события в любом случае остаются в бизнес-логе (даже в vacation mode)

## Триггеры: Tracking Points (REQ-NOTIF-007)

| Настройка | Описание |
|-----------|----------|
| Транспортные режимы | Все или только выбранные |
| STY-коды | Диапазон от/до (например, STY0001–STY0100) |
| location + tracking одновременно | Одно уведомление, а не два |

**Алерт при просроченном подтверждении:** если дата Pickup или Delivery прошла, а статус всё ещё PLANNED — уведомление: _«`<Shipment Name>` collect/delivery isn't confirmed yet but expected»_.

## Уведомления Chat (REQ-NOTIF-008)

Чат создаётся на уровне каждого объекта (slot, shipment, booking, etc.) с одной или несколькими сторонами.

| Настройка | Описание |
|-----------|----------|
| In-app | Настраивается для каждого чата по объекту |
| Email | Настраивается для каждого чата по объекту |
| /mute | Отписаться от конкретного чата |
| /follow | Подписаться на конкретный чат |

**Умолчания по уровню:**
- BOOKER: notify in-app = YES
- GROUP: notify in-app = YES
- ACCOUNT: notify in-app = NO

Уведомления из чата ретро-консолидации: назначение SR в группу, удаление, пересчёт с детализацией стоимостей.

## Гранулярность на уровне объекта (REQ-NOTIF-009)

На уровне каждого объекта доступны кнопки:
- **«Don't notify me anymore for this object»**
- **«Notify me for this object»**

Переопределяет глобальные настройки пользователя. Аналог /mute /follow из Slack.

## Первая волна реализации (REQ-NOTIF-010)

| Приоритет | Объекты |
|-----------|---------|
| Волна 1 | BOOKING, TRACKING, SLOT, CHAT (TD-359) |
| Волна 2 | INVOICE, CLAIM, и другие |

**Порядок реализации:**
1. Бизнес-лог (slicing по SHIPMENT, SHIPMENT REQUEST, SLOTS)
2. Редизайн NOTIFICATION (app/mail/none) с расширенным выбором пользователя
3. Полноценная поддержка multi-account
