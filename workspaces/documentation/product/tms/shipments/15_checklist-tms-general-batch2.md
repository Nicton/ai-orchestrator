## REQ-MC: Мульти-контейнер (Multi Container)

### REQ-MC-001 — Условие создания мульти-контейнера
**Приоритет:** P0 | **Источник:** 2020 02 - Multi Container

**Проверки:**
- [ ] Мульти-контейнер создаётся только при типе груза FCL или FTL и Qty > 1
- [ ] При Qty = 1 мульти-контейнерный режим не активируется
- [ ] Поддерживается альтернативный вариант: несколько контейнеров забронированы в рамках одного бронирования
- [ ] При создании бронирования до подтверждения интерфейс идентичен стандартному
- [ ] Стоимость управляется на уровне бронирования, а не на уровне отдельных контейнеров

---

### REQ-MC-002 — Именование контейнеров при подтверждении бронирования
**Приоритет:** P0 | **Источник:** 2020 02 - Multi Container

**Проверки:**
- [ ] При подтверждении бронирования система предлагает задать имя/суффикс каждой отправке
- [ ] Перевозчик может указать номер контейнера в поле имени
- [ ] Каждый контейнер получает уникальный суффикс (0001, 0002… 000N)
- [ ] Суффиксы отображаются в экране трекинга рядом с именем отправки
- [ ] После подтверждения пакинг-лист не может быть изменён на уровне бронирования

---

### REQ-MC-003 — Экран бронирования: список контейнеров
**Приоритет:** P1 | **Источник:** 2020 02 - Multi Container

**Проверки:**
- [ ] После подтверждения в средней колонке отображается список всех отправок с суффиксами
- [ ] Для каждой отправки показаны: дата отправки, дата прибытия, статус трекинга
- [ ] Доступна кнопка "See all containers activity"
- [ ] Чат имеет переключатель: показать сообщения всех контейнеров / только уровень бронирования
- [ ] Переключатель чата не меняет список подписчиков, только фильтрует сообщения

---

### REQ-MC-004 — Управление метаданными для нескольких контейнеров
**Приоритет:** P1 | **Источник:** 2020 02 - Multi Container

**Проверки:**
- [ ] При добавлении метаданных система предлагает выбор: "все контейнеры" или список выбранных
- [ ] Отображается "Update Metadata on all containers" с чекбоксами для каждого контейнера
- [ ] Пользователь может выбрать отдельные контейнеры через мультивыбор
- [ ] Доступны кнопки "select all" / "unselect all"
- [ ] Изменения метаданных применяются только к выбранным контейнерам

---

### REQ-MC-005 — Загрузка документов на несколько контейнеров
**Приоритет:** P1 | **Источник:** 2020 02 - Multi Container

**Проверки:**
- [ ] При загрузке документа появляется диалог "Share documents on"
- [ ] Варианты: "This container only" / "multiple containers"
- [ ] При выборе "multiple containers" отображается список с чекбоксами
- [ ] Поддерживаются "select all" и "unselect all"
- [ ] Документ сохраняется для каждого выбранного контейнера отдельно

---

### REQ-MC-006 — Трекинг: обновление точек трекинга для нескольких контейнеров
**Приоритет:** P1 | **Источник:** 2020 02 - Multi Container

**Проверки:**
- [ ] При подтверждении доставки система предлагает выбор: "all containers" или список выбранных
- [ ] Пользователь может перейти к другому контейнеру из текущего экрана
- [ ] Отображается счётчик связанных контейнеров ("5 other containers on this booking")
- [ ] При клике на счётчик раскрывается список всех контейнеров с их статусами
- [ ] Из списка можно перейти к любому контейнеру

---

### REQ-MC-007 — Инвойсинг мульти-контейнерных бронирований
**Приоритет:** P1 | **Источник:** 2020 02 - Multi Container

**Проверки:**
- [ ] Мульти-контейнерное бронирование отображается как одна строка в реестре инвойсов
- [ ] При раскрытии показывается список отправок с их статусами
- [ ] Указано количество забронированных контейнеров
- [ ] Для FCL/FTL грузов в пакинг-листе не отображаются размеры единиц
- [ ] При наличии N контейнеров даты группируются (не показываются N отдельных строк дат)

---

### REQ-MC-008 — Отмена отдельных контейнеров
**Приоритет:** P2 | **Источник:** 2020 02 - Multi Container

**Проверки:**
- [ ] Отмена отдельного контейнера выполняется через раздел TRACKING
- [ ] Отмена одного контейнера не отменяет всё бронирование
- [ ] Статус отменённого контейнера обновляется в списке бронирования
- [ ] Инвойс корректно отражает фактическое количество доставленных контейнеров

## REQ-SEA: Морские перевозки — данные судна (SeaFreight Ship Data)

### REQ-SEA-001 — Структура данных морского перехода (SeaLeg)
**Приоритет:** P0 | **Источник:** 2023 01 - SeaFreight - Ship Data

**Проверки:**
- [ ] Каждая морская отправка поддерживает 1 BL (Bill of Lading)
- [ ] Для одной отправки можно добавить N морских переходов (SeaLeg)
- [ ] Каждый SeaLeg содержит: Ship Name, Ship IMO (7 цифр), Company IMO, Company Name
- [ ] Каждый SeaLeg содержит: Port of Loading (ISO), ETD, Port of Arrival (ISO), ETA
- [ ] Поле BL ID хранится на уровне отправки, а не на уровне перехода

---

### REQ-SEA-002 — Управление справочником морских перевозчиков
**Приоритет:** P1 | **Источник:** 2023 01 - SeaFreight - Ship Data

**Проверки:**
- [ ] Система хранит центральный справочник морских перевозчиков на уровне Shiptify
- [ ] Справочник включает ~20-30 крупнейших перевозчиков (95% мировых потоков)
- [ ] При выборе перевозчика отображается логотип компании
- [ ] Поле перевозчика — ограниченный список, не свободный ввод
- [ ] Компания может быть добавлена как перевозчик в STY с признаком "sea carrier"

---

### REQ-SEA-003 — Добавление данных о судне (Add Ship Info)
**Приоритет:** P0 | **Источник:** 2023 01 - SeaFreight - Ship Data

**Проверки:**
- [ ] Кнопка "Add Ship Info" доступна для всех морских отправок (MC и не-MC)
- [ ] Форма содержит поля: Carrier, Ship Name, Port From, ETD, Port To, ETA, BL ID
- [ ] Поддерживается добавление нескольких переходов через "+ add transhipment"
- [ ] Для не-MC отправок кнопка действия — "UPDATE" (закрывает модал)
- [ ] Для MC отправок кнопка действия — "NEXT" (переходит к следующему шагу)

---

### REQ-SEA-004 — Обновление данных судна для группы контейнеров
**Приоритет:** P1 | **Источник:** 2023 01 - SeaFreight - Ship Data

**Проверки:**
- [ ] При MC или связанных отправках отображается выбор области применения
- [ ] Варианты: "All containers", "Selection", "Unassigned", "This container only"
- [ ] "Unassigned" означает контейнеры без назначенного SeaLeg
- [ ] При выборе "Selection" пользователь видит список контейнеров с текущими данными SeaLeg
- [ ] После обновления данные SeaLeg синхронизируются для всех выбранных контейнеров

---

### REQ-SEA-005 — Отображение данных судна в UI
**Приоритет:** P1 | **Источник:** 2023 01 - SeaFreight - Ship Data

**Проверки:**
- [ ] Данные судна отображаются в формате: "Ship Name | PORT_FROM Date > PORT_TO Date"
- [ ] Используются ISO-коды портов (FRHAV, SISGP, DUBAI и т.д.)
- [ ] При наличии нескольких SeaLeg все переходы отображаются последовательно
- [ ] Информация доступна через mouseover на логотип судоходной компании
- [ ] Данные судна доступны через API и могут использоваться в интеграциях

---

### REQ-SEA-006 — Передача документов в контексте Sea Freight + MC
**Приоритет:** P1 | **Источник:** 2023 01 - SeaFreight - Ship Data

**Проверки:**
- [ ] Документы для Sea Freight MC можно делиться по "Same Ship" (одно судно)
- [ ] Варианты распространения: "This shipment", "All shipments", "Selection", "Same Ship"
- [ ] При выборе "Same Ship" документ применяется ко всем контейнерам на том же судне
- [ ] Категория "Unassigned" не отображается в диалоге передачи документов
- [ ] Компонент выбора контейнеров показывает текущий SeaLeg каждого контейнера

## REQ-DOC: Документооборот (Documents Workflow)

### REQ-DOC-001 — Настройка уведомлений о документах на уровне пользователя
**Приоритет:** P0 | **Источник:** 2024 11 - Documents workflow

**Проверки:**
- [ ] Пользователь может создать один или несколько workflow уведомлений
- [ ] Workflow настраивается через профиль пользователя (отдельная вкладка Notifications)
- [ ] Страница настройки доступна только через профиль, не через общие настройки
- [ ] При дублировании условий уведомления пользователь не получает двойное оповещение
- [ ] Workflow работает для API-потоков и для ручных загрузок

---

### REQ-DOC-002 — Типы документов в workflow
**Приоритет:** P0 | **Источник:** 2024 11 - Documents workflow

**Проверки:**
- [ ] Список типов документов ограничен теми, что включены на уровне аккаунта
- [ ] Пользователь может добавить несколько типов документов в один workflow
- [ ] Пользователь может удалить отдельный тип документа из workflow
- [ ] Необходимо добавить минимум один тип документа для сохранения workflow
- [ ] Примеры поддерживаемых типов: Proof of Delivery, Bill of Lading, Commercial Invoice

---

### REQ-DOC-003 — Условия триггера workflow уведомлений
**Приоритет:** P0 | **Источник:** 2024 11 - Documents workflow

**Проверки:**
- [ ] Условие "Когда я владелец объекта" (booking, shipment, TR, slots) — доступно
- [ ] Условие "Все потоки, созданные API на этом аккаунте" — доступно
- [ ] Условие "Все потоки, созданные вручную на этом аккаунте" — доступно
- [ ] Условие "Все потоки на этом аккаунте" — доступно (default)
- [ ] Для мультиаккаунтных пользователей настройки применяются к каждому аккаунту отдельно

---

### REQ-DOC-004 — Действия, triggering уведомление
**Приоритет:** P0 | **Источник:** 2024 11 - Documents workflow

**Проверки:**
- [ ] Действие "Added" (документ загружен) — включено по умолчанию
- [ ] Действие "Removed" (документ удалён или отменён) — настраивается
- [ ] Действие "Visibility changed" (изменена видимость документа) — настраивается
- [ ] Пользователь может выбрать одно или несколько действий
- [ ] Как минимум одно действие должно быть выбрано

---

### REQ-DOC-005 — Географический периметр уведомлений
**Приоритет:** P1 | **Источник:** 2024 11 - Documents workflow

**Проверки:**
- [ ] Поддерживается фильтр FROM / TO по: Country, LogZone
- [ ] Настройка периметра опциональна; при отсутствии — применяется ко всем
- [ ] Пример корректного использования: уведомлять только о POD из FR95 в Испанию
- [ ] Структура периметра аналогична логике транспортного плана
- [ ] Периметр отображается в строке сводки workflow

---

### REQ-DOC-006 — Канал доставки уведомлений
**Приоритет:** P1 | **Источник:** 2024 11 - Documents workflow

**Проверки:**
- [ ] Вариант "Notification" (in-app) — выбран по умолчанию
- [ ] Вариант "Mail" — доступен для выбора
- [ ] Вариант "Notification & Mail" — доступен для выбора
- [ ] Настройка канала независима для каждого workflow
- [ ] Мультиаккаунтные пользователи получают уведомления для всех затронутых аккаунтов

---

### REQ-DOC-007 — Список workflow и управление ими
**Приоритет:** P1 | **Источник:** 2024 11 - Documents workflow

**Проверки:**
- [ ] Workflow отображается в виде строк: документы, счётчик, условие, действия, периметр, канал
- [ ] Клик на строку открывает workflow для просмотра и редактирования
- [ ] Пользователь может удалить workflow
- [ ] Доступна кнопка "Add a workflow" для создания нового
- [ ] Иконки аккаунтов отображаются в строке для идентификации

---

### REQ-DOC-008 — Выбор аккаунтов для workflow
**Приоритет:** P1 | **Источник:** 2024 11 - Documents workflow

**Проверки:**
- [ ] По умолчанию предвыбирается основной аккаунт пользователя
- [ ] Пользователь может добавить другие аккаунты из MULTI ACC setup (BO + CONSTELLATION)
- [ ] Если пользователь привязан только к одному аккаунту — список аккаунтов скрыт
- [ ] Выбранные аккаунты отображаются в сводке workflow
- [ ] Workflow применяется независимо для каждого выбранного аккаунта

---

### REQ-DOC-009 — Уровни доступа к документам в групповых отправках
**Приоритет:** P0 | **Источник:** 2023 02 - STY 3.0

**Проверки:**
- [ ] Уровень "Private": только мой аккаунт видит документ
- [ ] Уровень "Limited": только Shipper и Carrier видят документ
- [ ] Уровень "Public": все аккаунты, имеющие доступ к отправке, видят документ
- [ ] Уровень "Specific": пользователь выбирает конкретные аккаунты вручную
- [ ] При добавлении нового аккаунта к отправке позднее — Public-документы автоматически становятся ему доступны

---

### REQ-DOC-010 — Передача документов по типу группировки
**Приоритет:** P1 | **Источник:** 2023 02 - STY 3.0

**Проверки:**
- [ ] Multi Container: варианты "This shipment", "All shipments", "Selection", "Same Ship"
- [ ] Grouped Departure: варианты "This shipment", "All shipments", "Selection", "Same Destination"
- [ ] Grouped Delivery: варианты "This shipment", "All shipments", "Selection", "Same Pick-up"
- [ ] Grouped Shipments: варианты "This shipment", "All shipments", "Selection"
- [ ] Milkrun: варианты "This shipment", "All shipments", "Selection"

---

### REQ-DOC-011 — Уведомления при загрузке документов в группе
**Приоритет:** P1 | **Источник:** 2023 02 - STY 3.0

**Проверки:**
- [ ] Уведомления в чате показываются только аккаунтам, которым документ был предоставлен
- [ ] Вместо полного списка пользователей отображается числовой тег (например, "12")
- [ ] Mouseover на тег показывает список предоставленных аккаунтов и уведомлённых пользователей
- [ ] Тег визуально показывает уровень доступа: private / limited / specific (public — без тега)
- [ ] Уведомление не отправляется повторно если пользователь уже был уведомлён вручную

## REQ-STY: STY 3.0 — двусторонние аккаунты и Transport Request

### REQ-STY-001 — Двусторонняя модель аккаунта (Carrier + Shipper)
**Приоритет:** P0 | **Источник:** 2023 02 - STY 3.0

**Проверки:**
- [ ] Любой аккаунт Shiptify может быть одновременно Carrier и Shipper
- [ ] Аккаунт имеет свойства: "Can Sell: YES/NO" и "Can Buy: YES/NO"
- [ ] Каждый пользователь имеет свойства: "Can Sell: YES/NO" и "Can Buy: YES/NO"
- [ ] Поддерживаются цепочки: A покупает у B, B покупает у C, C покупает у D
- [ ] Авторизованный пользователь может переключаться между режимами BUY/SALES

---

### REQ-STY-002 — Новый объект: Transport Request (QR)
**Приоритет:** P0 | **Источник:** 2023 02 - STY 3.0

**Проверки:**
- [ ] QR — объект с точкой A, точкой B и списком груза C
- [ ] QR может быть Multi Container (аналогично Booking)
- [ ] QR поддерживает группировку (Milkrun, Grouped Departure, Grouped Delivery, Grouped)
- [ ] QR может быть назначен на BK и отвязан от BK
- [ ] QR может быть назначен на SH, отвязан от SH и переназначен на другой SH

---

### REQ-STY-003 — Жизненный цикл Transport Request
**Приоритет:** P0 | **Источник:** 2023 02 - STY 3.0

**Проверки:**
- [ ] QR имеет статусы: Pending, Assigned, Cancelled
- [ ] QR связан с: Requester (Account, обязательно), Shipper (Buy Account, обязательно), Carrier (Sell Account, опционально)
- [ ] QR назначен на BK → может создать SH → SH может быть отменён
- [ ] В рамках одного BK может быть N QR
- [ ] N QR могут быть назначены на один BK или разные BK

---

### REQ-STY-004 — Дублирование чатов при создании SH из QR
**Приоритет:** P1 | **Источник:** 2023 02 - STY 3.0

**Проверки:**
- [ ] При создании SH из QR создаётся новый чат на уровне SH для каждого чата QR (с историей)
- [ ] При создании SH из QR создаётся новый чат на уровне SH для каждого чата BK (с историей)
- [ ] Сообщение на уровне подтверждённого QR копируется во все связанные SH
- [ ] Сообщение на уровне PSH копируется в связанный SH и QR
- [ ] Чат привязывается к: PSH, QR, BK или SH

---

### REQ-STY-005 — Навигация BUY / SALES
**Приоритет:** P1 | **Источник:** 2023 02 - STY 3.0

**Проверки:**
- [ ] При Can Sell = YES пользователь видит carrier-интерфейс: Booking, Grouping, Tracking, Invoicing, Claims, Multivision, Dashboard
- [ ] Переключатель BUY/SALES доступен авторизованным пользователям
- [ ] Разделы: BUY side — видит бронирования "to" нужным аккаунтам; SALES side — видит бронирования "from"
- [ ] Новый раздел "Requests" (EN) / "Demandes" (FR) создаётся отдельно от "Booking"
- [ ] Французское название "Booking" переименовывается из "Demandes" в "Booking"

---

### REQ-STY-006 — Общие настройки MD и документов по умолчанию
**Приоритет:** P1 | **Источник:** 2023 02 - STY 3.0

**Проверки:**
- [ ] Настройки по умолчанию для метаданных и документов задаются на уровне MD и Doc
- [ ] Настройки применяются при создании BK и при создании SH
- [ ] Подписчики чата BK формируются при создании BK по персональным настройкам пользователей
- [ ] Подписчики чата SH формируются при создании SH по персональным настройкам пользователей
- [ ] Изменение настроек не затрагивает уже существующие чаты

---

### REQ-STY-007 — Spectators в контексте BK и QR
**Приоритет:** P1 | **Источник:** 2023 02 - STY 3.0

**Проверки:**
- [ ] Spectators добавляются на уровне BK
- [ ] Spectator имеет доступ ко всем публичным группам чата в каждом QR
- [ ] Spectator имеет ограниченный доступ к информации QR (средняя колонка, ставки)
- [ ] Группы чата: Public, Private (только моя команда), Private Carrier/Shipper
- [ ] Любой пользователь аккаунта может иметь доступ к любой группе чата, где аккаунт активирован

---

### REQ-STY-008 — Управление уведомлениями пользователя
**Приоритет:** P1 | **Источник:** 2023 02 - STY 3.0

**Проверки:**
- [ ] Создаётся отдельная вкладка "Notifications" в профиле пользователя
- [ ] На carrier-стороне доступен раздел "Customers" для настройки фильтра клиентов
- [ ] Пользователь может активировать/деактивировать уведомления по конкретным клиентам (Galaxies)
- [ ] Кнопка "Notify" заменяет логику на "SEND ONLY CHAT MESSAGE NOTIFICATIONS"

## REQ-EO: Expected Order (EO) и Collaborative Order

### REQ-EO-001 — Создание объекта EO (Expected Order)
**Приоритет:** P0 | **Источник:** 2024 02 - EO new object + Collaborative Order

**Проверки:**
- [ ] EO имеет уникальный UID на уровне PML/Supplier ID
- [ ] EO содержит поля: EO_Name, EO_Zone, EO_WH_Customer_ID, EO_Supplier_ID, EO_Supplier_Name, EO_Supplier_Email
- [ ] EO содержит: EO_Packing_List_Qty, EO_Packing_List_Cargo_Type, EO_Content_Parcels, EO_Content_Units
- [ ] EO имеет статусы: Open, Closed, Cancelled, Ongoing (автоматический)
- [ ] EO содержит: EO_Delivery_Min_date, EO_Delivery_Max_date, EO_Creation_Date, EO_Tag, EO_Auto_Close

---

### REQ-EO-002 — Связь EO с заказами и слотами
**Приоритет:** P0 | **Источник:** 2024 02 - EO new object + Collaborative Order

**Проверки:**
- [ ] WH Customer определяется на уровне ORDER, а не на уровне SLOT
- [ ] Один ORDER может быть назначен на N слотов (связь N:N через EO)
- [ ] При бронировании слота создаются ORDER из матрицы EO
- [ ] EO хранит все данные, которые ранее хранились в SLOT ORDER
- [ ] Загрузка XLS-листа обновляет базу EO, а не базу ORDER

---

### REQ-EO-003 — Процесс бронирования слота через EO
**Приоритет:** P0 | **Источник:** 2024 02 - EO new object + Collaborative Order

**Проверки:**
- [ ] При бронировании слота система ищет соответствующие EO вместо DOCK ORDER
- [ ] Carrier/Supplier выбирает, что хочет доставить: паллеты, коробки, единицы
- [ ] Логическая проверка соответствия не выполняется (доставщик знает лучше)
- [ ] После бронирования создаётся ORDER, привязанный к слоту и к родительскому EO
- [ ] Зона слота по умолчанию берётся из EO, но может быть изменена без влияния на EO

### REQ-EO-004 — Управление статусами EO из PML аккаунта

- [ ] Заказы в статусе ASSIGNED и DELIVERED копируются в EO со статусом = CLOSED

## REQ-LABEL: Генерация штрихкодов и мульти-лейблы

### REQ-LABEL-001 — Включение SSCC на уровне аккаунта
**Приоритет:** P0 | **Источник:** 2025 06 - Barcode generation and Multi Label triggering

**Проверки:**
- [ ] В настройках покупателя (buyer) доступна опция "Enable SSCC on labels" (по умолчанию OFF)
- [ ] При включении PREFIX становится обязательным полем (ровно 7 цифр)
- [ ] PREFIX должен быть уникальным в базе данных Shiptify
- [ ] При включении SSCC метки генерируются автоматически на уровне единицы, а не по запросу
- [ ] Метки сохраняются в менеджере документов

---

### REQ-LABEL-002 — Структура SSCC кода
**Приоритет:** P0 | **Источник:** 2025 06 - Barcode generation and Multi Label triggering

**Проверки:**
- [ ] Структура SSCC: Extension digit (1) + Company prefix (7) + Serial reference (9) + Check digit (1) = 18 символов
- [ ] Company prefix управляется вручную на уровне аккаунта и уникален в базе
- [ ] Serial reference — последовательный счётчик на уровне аккаунта
- [ ] Check digit рассчитывается по алгоритму modulo 10 от первых 17 цифр
- [ ] Вдохновлено стандартом GS1-128, но не требует регистрации в GS1

---

### REQ-LABEL-003 — Trigger генерации штрихкодов на уровне перевозчика
**Приоритет:** P0 | **Источник:** 2025 06 - Barcode generation and Multi Label triggering

**Проверки:**
- [ ] Если SSCC включён на уровне аккаунта, для каждого активного перевозчика появляется опция "Generate barcodes" (по умолчанию NO)
- [ ] Trigger активируется при: SSCC включён на аккаунте + Generate barcodes включён для перевозчика
- [ ] SSCC генерируется в момент VALIDATING BOOKED SR/PSH (между PSH→SH)
- [ ] Сгенерированный SSCC должен быть включён в webhook "create shipment" для перевозчика
- [ ] Режим генерации: мульти-лейбл вместо единственной метки если условия выполнены

---

### REQ-LABEL-004 — Хранение SSCC в структуре данных
**Приоритет:** P0 | **Источник:** 2025 06 - Barcode generation and Multi Label triggering

**Проверки:**
- [ ] SSCC хранится в поле FREIGHT UNIT KEY на уровне контента
- [ ] При Quantity > 1 и активном trigger контент автоматически разбивается на строки по 1 единице
- [ ] Каждая строка контента с Qty=1 получает свой уникальный SSCC
- [ ] Webhook возвращает freight_unit_key для каждой строки контента отдельно
- [ ] SSCC доступен через API, UI и Elastic Search

---

### REQ-LABEL-005 — Управление изменениями пакинг-листа при SSCC
**Приоритет:** P1 | **Источник:** 2025 06 - Barcode generation and Multi Label triggering

**Проверки:**
- [ ] При обновлении пакинг-листа — метки регенерируются (сохраняется существующая логика)
- [ ] SSCC не перегенерируются при обновлении пакинг-листа — сохраняются существующие
- [ ] Добавление новой единицы: новый SSCC для новой строки, существующие не меняются
- [ ] Удаление единицы: удаляется только SSCC удалённой строки
- [ ] Все метки хранятся в одном PDF (аналогично текущему скачиванию мульти-лейблов)

---

### REQ-LABEL-006 — Совместимость с workflow отправки по почте
**Приоритет:** P1 | **Источник:** 2025 06 - Barcode generation and Multi Label triggering

**Проверки:**
- [ ] Мульти-лейблы с SSCC работают с workflow "Send by Mail" без дополнительных изменений
- [ ] Оптимальный случай: мульти-лейбл генерируется автоматически по той же логике, что и единственная метка
- [ ] Одиночная метка (Single label) пересоздаётся в менеджере документов
- [ ] Клик "Generate N labels" скачивает лист меток без сохранения в Shiptify
- [ ] В будущем должна поддерживаться совместимость с форматом ZPL 10x10

## REQ-VIS: Управление визитами и водителями

### REQ-VIS-001 — Разделение ожидаемых и проверенных данных водителя
**Приоритет:** P0 | **Источник:** 2024 01 - VISITS drivers

**Проверки:**
- [ ] Данные водителя разделены на два независимых набора: Expected Driver data и Checked Driver data
- [ ] Данные грузовика разделены на два независимых набора: Expected Truck data и Checked Truck data
- [ ] Все существующие данные из API / SLOTIFY / SLOTBOOK обновляют только EXPECTED данные
- [ ] UI обеспечивает процесс заполнения CHECKED данных
- [ ] Expected и Checked данные хранятся независимо в базе данных

---

### REQ-VIS-002 — UI блока водителя в карточке визита
**Приоритет:** P1 | **Источник:** 2024 01 - VISITS drivers

**Проверки:**
- [ ] Блок водителя оформлен светло-серым фоном для визуальной читаемости
- [ ] Номер документа водителя скрывает последние 3 цифры (заменяется на ***)
- [ ] Кнопка SMS отображается только если указан номер телефона
- [ ] При отсутствии данных — выводится список недостающих полей (Missing data)
- [ ] Поле TRUCK имеет максимальную ширину для 10 символов (номера имеют схожую длину)

---

### REQ-VIS-003 — Идентификация заблокированных водителей
**Приоритет:** P0 | **Источник:** 2024 01 - VISITS drivers

**Проверки:**
- [ ] При идентификации заблокированного водителя отображается светло-оранжевый блок (Expected)
- [ ] При заполнении CHECKED данных с заблокированным водителем отображается тёмно-оранжевый/красный блок
- [ ] Пользователь видит предупреждение с номером (#XX) заблокированного водителя
- [ ] Пользователь может зарегистрировать заблокированного водителя и затем отказать в доступе на уровне VISIT
- [ ] Случай заблокированного водителя является критическим инцидентом

---

### REQ-VIS-004 — Процесс заполнения CHECKED Driver данных
**Приоритет:** P0 | **Источник:** 2024 01 - VISITS drivers

**Проверки:**
- [ ] Только ADMIN пользователь видит модал обновления EXPECTED DRIVER
- [ ] Не-admin пользователи пропускают шаг обновления EXPECTED данных (security)
- [ ] Модал CHECKED DRIVER предзаполняется из EXPECTED DRIVER данных
- [ ] Поля документа предзаполняются с на 3 цифры меньше, чем в EXPECTED (пользователь вводит полный номер)
- [ ] Кнопка "Save in Database" доступна только в разделе CHECKED DRIVER

---

### REQ-VIS-005 — Валидация CHECKED данных
**Приоритет:** P0 | **Источник:** 2024 01 - VISITS drivers

**Проверки:**
- [ ] Если введённые данные короче, чем EXPECTED данные — отображается красная рамка
- [ ] Если введённые данные равны EXPECTED — отображается зелёный индикатор
- [ ] Если введённые данные отличаются от EXPECTED — отображается красный индикатор
- [ ] Если EXPECTED номер документа пуст — принимается любое значение
- [ ] CHECKED данные для грузовика проверяют как LICENCE PLATE, так и TRAILER

---

### REQ-VIS-006 — Логирование расхождений водителя и грузовика
**Приоритет:** P1 | **Источник:** 2024 01 - VISITS drivers

**Проверки:**
- [ ] При несоответствии данных водителя с ожидаемыми: Driver_data_gap = 1 на уровне VISIT
- [ ] При несоответствии данных грузовика с ожидаемыми: Truck_data_gap = 1 на уровне VISIT
- [ ] Флаги gap позволяют в будущем фильтровать VISIT с расхождениями
- [ ] Пользователь видит предупреждение с информацией о расхождении перед сохранением
- [ ] После сохранения CHECKED данных — блок визита обновляется синим фоном

---

### REQ-VIS-007 — Сохранение водителя в базе данных
**Приоритет:** P1 | **Источник:** 2024 01 - VISITS drivers

**Проверки:**
- [ ] Водитель идентифицируется по комбинации: тип документа + номер документа
- [ ] Если водитель с такими данными не существует в DB — создаётся новая запись
- [ ] После сохранения Checked Driver отображаются данные с синим фоном (verified)
- [ ] Логика сохранения в базу данных не применяется для EXPECTED данных
- [ ] Данные CHECKED DRIVER хранятся независимо от EXPECTED DRIVER
