## TMS BOOKING


### REQ-BOOK-001 — Создание заявки — Standard Creation (SC)

**Приоритет:** P0 | **Источник:** 2026 01 - Flow Types - Google Slides.txt, Page 4


**Проверки:**
- [ ] Шиппер может вручную создать заявку (SC) для режимов Air/Sea/Road/Express/Parcel
- [ ] CSW открывается пустым для ручного заполнения всех полей
- [ ] После создания SC доступна опция сохранить как шаблон
- [ ] SC не требует предварительного шаблона или внешней ссылки
- [ ] Создание заявки недоступно для роли Carrier
- [ ] Пользователь видит ошибку валидации при попытке отправить неполный SC
- [ ] Созданная SC отображается в списке со статусом new/draft

### REQ-BOOK-002 — Создание заявки из шаблона (Template Creation — TC)

**Приоритет:** P0 | **Источник:** 2026 01 - Flow Types - Google Slides.txt, Page 5


**Проверки:**
- [ ] Шиппер может выбрать шаблон из базы шаблонов и создать заявку (TC)
- [ ] CSW открывается предзаполненным данными шаблона
- [ ] Пользователь может редактировать любое поле предзаполненного CSW
- [ ] TC доступно для режимов Air/Sea/Road/Express/Parcel
- [ ] Шаблон с некорректными/устаревшими данными всё равно позволяет создать заявку после ручной коррекции
- [ ] Созданная TC корректно сохраняет ссылку на исходный шаблон

### REQ-BOOK-003 — Создание заявки по Token Template (TT) — без авторизации

**Приоритет:** P1 | **Источник:** 2026 01 - Flow Types - Google Slides.txt, Page 6


**Проверки:**
- [ ] Внешний партнёр шиппера открывает уникальную ссылку и создаёт заявку без логина
- [ ] CSW для TT предзаполнен полями шаблона, нельзя редактировать заблокированные поля
- [ ] Созданная через TT заявка попадает в статус DRAFT или сразу направляется перевозчику согласно настройкам
- [ ] Попытка создать TT с просроченной/невалидной ссылкой — показывается ошибка
- [ ] TT поддерживает все режимы: Air/Sea/Road/Express/Parcel
- [ ] Идентификатор внешнего партнёра сохраняется в созданной заявке
- [ ] Нельзя использовать один TT-токен более одного раза, если настроено однократное использование

### REQ-BOOK-004 — Book For Me (BFM) — внешний партнёр с бесплатным аккаунтом

**Приоритет:** P1 | **Источник:** 2026 01 - Flow Types - Google Slides.txt, Page 7


**Проверки:**
- [ ] Внешний партнёр шиппера с бесплатным Shiptify-аккаунтом создаёт заявку через BFM
- [ ] Заявка BFM попадает на сторону шиппера в статусе DRAFT или направляется перевозчику
- [ ] Статус заявки BFM виден всем участникам совместно (коллаборативный статус)
- [ ] Трассировка: видно кто и когда создал заявку через BFM
- [ ] Без авторизации BFM-процесс недоступен
- [ ] BFM работает для режимов Air/Sea/Road/Express/Parcel

### REQ-BOOK-005 — Повторное создание заявки (Repeat Creation — RC)

**Приоритет:** P1 | **Источник:** 2026 01 - Flow Types - Google Slides.txt, Page 8


**Проверки:**
- [ ] Кнопка Repeat доступна на странице существующей заявки/отправки
- [ ] CSW открывается с предзаполненными From, To и Cargo из исходной заявки
- [ ] Дата отправки и прибытия не копируется — требует нового ввода
- [ ] Repeat доступно для режимов Air/Sea/Road/Express/Parcel
- [ ] Пользователь может изменить любое предзаполненное поле перед отправкой
- [ ] Созданная RC не имеет жёсткой связи с оригиналом (независимая запись)

### REQ-BOOK-006 — Forward Creation (FW) — создание продолжения маршрута

**Приоритет:** P2 | **Источник:** 2026 01 - Flow Types - Google Slides.txt, Page 9


**Проверки:**
- [ ] Кнопка Forward доступна на странице заявки A→B
- [ ] CSW открывается с From=B и Cargo=C (откуда предыдущая заявка доставляет)
- [ ] Поле To и даты требуют ручного ввода
- [ ] Forward работает для Air/Sea/Road/Express/Parcel
- [ ] Созданная FW связана с родительской заявкой (ссылка mother→daughter)

- [ ] Даты требуют нового ввода
- [ ] Throwback работает для Air/Sea/Road/Express/Parcel
- [ ] Созданная TB связана с родительской заявкой (ссылка mother→daughter)

### REQ-BOOK-009 — Создание заявки из PDF через AI (PDFAI)

**Приоритет:** P1 | **Источник:** 2026 01 - Flow Types - Google Slides.txt, Page 12


**Проверки:**
- [ ] Пользователь загружает PDF-файл и система распознаёт поля заявки
- [ ] Распознанные значения отображаются как предложения с возможностью подтверждения или корректировки
- [ ] Поля, которые система не смогла распознать, остаются пустыми и требуют ручного ввода
- [ ] PDFAI доступно для режимов Air/Sea/Road/Express/Parcel
- [ ] Нечитаемый или нерелевантный PDF не вызывает краша — показывается понятная ошибка
- [ ] После подтверждения полей создаётся стандартная заявка с сохранённым источником PDFAI

### REQ-BOOK-010 — Создание заявки из текста через AI (TXTAI)

**Приоритет:** P1 | **Источник:** 2026 01 - Flow Types - Google Slides.txt, Page 13


**Проверки:**
- [ ] Пользователь вставляет произвольный текст и система распознаёт поля заявки
- [ ] Распознанные значения отображаются как предложения, каждое можно принять или изменить
- [ ] Неопознанные поля остаются пустыми и помечаются как требующие заполнения
- [ ] TXTAI поддерживает режимы Air/Sea/Road/Express/Parcel
- [ ] Пустой или бессмысленный текст не вызывает ошибки сервера — показывается UX-ошибка
- [ ] Созданная заявка сохраняет тип источника TXTAI

### REQ-BOOK-011 — Создание заявки из Transport Request (TRC)

**Приоритет:** P1 | **Источник:** 2026 01 - Flow Types - Google Slides.txt, Page 14


**Проверки:**
- [ ] Заявка создаётся из TR, ожидающего подтверждения на аккаунте шиппера
- [ ] TRC доступна только для аккаунтов типа Buy&Sell (не для чистого Shipper или Carrier)
- [ ] Упаковочный лист (packing list) TR переносится в созданную заявку
- [ ] Созданная заявка сохраняет ссылку на исходный TR для трассировки Sell→Buy
- [ ] Маржа рассчитывается на уровне TR и видна в созданной заявке
- [ ] Попытка создать TRC без привязки к TR — должна возвращать ошибку

### REQ-BOOK-012 — Direct Booking (DB) — выбор одного перевозчика

**Приоритет:** P0 | **Источник:** 2026 01 - Flow Types - Google Slides.txt, Page 17


**Проверки:**
- [ ] Шиппер выбирает одного перевозчика из панели и отправляет заявку
- [ ] Перевозчик получает уведомление и видит кнопки подтверждения Green/Red
- [ ] Перевозчик подтверждает заявку — статус SR переходит в confirmed
- [ ] Перевозчик отклоняет заявку — статус SR переходит в refused/declined
- [ ] DB доступен для режимов Air/Sea/Road
- [ ] Перевозчик, у которого нет доступа к заявке, не видит её в своём списке

### REQ-BOOK-013 — Quote Request (QR) — запрос котировок у нескольких перевозчиков

**Приоритет:** P0 | **Источник:** 2026 01 - Flow Types - Google Slides.txt, Page 20


**Проверки:**
- [ ] Шиппер может выбрать несколько перевозчиков одновременно для QR
- [ ] Каждый перевозчик отправляет котировку в стандартизированном формате внутри Shiptify
- [ ] Шиппер видит все котировки в едином интерфейсе сравнения
- [ ] После выбора котировки процесс продолжается как Direct Booking
- [ ] QR доступен для Air/Sea/Road
- [ ] Перевозчик может отправить только одну котировку на один QR (если не включена опция multiple quotes per carrier)
- [ ] QR без точных дат (quote without precise dates) должен сохраняться как черновик

### REQ-BOOK-014 — Automated Direct Booking (ADB) — полностью автоматическое бронирование

**Приоритет:** P1 | **Источник:** 2026 01 - Flow Types - Google Slides.txt, Page 18


**Проверки:**
- [ ] ERP/WMS отправляет данные напрямую в Shiptify, перевозчик уведомляется автоматически
- [ ] Перевозчик подтверждает или отклоняет заявку кнопкой Green/Red
- [ ] ADB работает без участия человека со стороны шиппера
- [ ] ADB доступен для Air/Sea/Road
- [ ] Ошибка API при ADB логируется и видна в business log с payload
- [ ] Transport plan и Rates Management влияют на автовыбор перевозчика при ADB

### REQ-BOOK-015 — Semi Automated Direct Booking (SADB) — авто-создание с ручным выбором перевозчика

**Приоритет:** P1 | **Источник:** 2026 01 - Flow Types - Google Slides.txt, Page 19


**Проверки:**
- [ ] ERP/WMS создаёт черновик заявки, шиппер проверяет данные и вручную выбирает перевозчика
- [ ] Шиппер может дополнить данные или прикрепить документы перед отправкой
- [ ] Только после выбора перевозчика шиппером заявка отправляется и перевозчик уведомляется
- [ ] SADB доступен для Air/Sea/Road
- [ ] Неполная заявка SADB остаётся в статусе DRAFT до ручного завершения
- [ ] Шиппер видит список SADB-черновиков и может фильтровать их

### REQ-BOOK-016 — First Responder (FR) — первый перевозчик с ценой ниже лимита получает бронь

**Приоритет:** P2 | **Источник:** 2026 01 - Flow Types - Google Slides.txt, Page 23


**Проверки:**
- [ ] Шиппер задаёт максимальную стоимость (публичную или приватную) и выбирает несколько перевозчиков
- [ ] Первый перевозчик, предложивший цену ниже максимума, автоматически получает заявку
- [ ] Остальные перевозчики уведомляются об отмене запроса после присвоения
- [ ] FR доступен для Air/Sea/Road
- [ ] Если ни один перевозчик не уложился в лимит за заданное время — заявка остаётся открытой
- [ ] Шиппер может видеть, сколько перевозчиков успело ответить до назначения победителя

### REQ-BOOK-018 — Отмена отправки — причина отмены (Reason Code)

**Приоритет:** P0 | **Источник:** 2026 01 - Replan reason code & cancel (emma) - Google Slides.txt, Pages 2-4


**Проверки:**
- [ ] При нажатии Cancel Shipment открывается модальное окно с выпадающим списком причин отмены
- [ ] Администратор аккаунта может настроить поле причины как обязательное, опциональное или скрытое
- [ ] Администратор аккаунта может настроить поле комментария как обязательное или опциональное
- [ ] Если для комментария включён минимальный размер текста — валидация блокирует отправку при несоблюдении
- [ ] Список причин отмены формируется на уровне аккаунта (кастомный per-account)
- [ ] Миграция: дефолтное значение — DO NOT REQUEST (причина не запрашивается)
- [ ] Пользователь без права отмены не видит кнопку Cancel Shipment

### REQ-BOOK-019 — Отмена отправки — хранение данных и аудит

**Приоритет:** P1 | **Источник:** 2026 01 - Replan reason code & cancel (emma) - Google Slides.txt, Page 8


**Проверки:**
- [ ] При отмене сохраняются: user_id, user_name, account_id, account_name, SH_ID, SR_ID, дата/время отмены
- [ ] Сохраняется выбранная причина отмены (reason_code_name) и свободный текст (comment)
- [ ] Данные отмены видны в чате как лог (reason code + comment)
- [ ] Причина отмены видна всем участникам с доступом к чату (carrier и shipper)
- [ ] Smart-list «Shipments с отменой» предоставляет экспорт данных по отменам
- [ ] API-отмена принимает cancellation_reason и comment без проверки account-настроек (mandatory/optional)

### REQ-BOOK-020 — Replanning tracking point — причина переноса (Replan Reason Code)

**Приоритет:** P1 | **Источник:** 2026 01 - Replan reason code & cancel (emma) - Google Slides.txt, Pages 6-7


**Проверки:**
- [ ] При replanning TP (после выбора новой даты/времени) показывается шаг с причиной переноса
- [ ] Администратор аккаунта настраивает поле причины переноса: обязательное/опциональное/не запрашивать
- [ ] Если настройка DO NOT REQUEST — шаг причины пропускается (поведение как сегодня)
- [ ] Список причин переноса формируется на уровне аккаунта с поддержкой свободного ввода (custom value)
- [ ] Причина переноса и комментарий сохраняются в базе вместе с: user_id, account_id, role (carrier/shipper/other), SH_ID, SR_ID, STY_CODE, начальная дата, новая дата, счётчик переносов для данного TP
- [ ] Общий счётчик переносов на уровне SH (сумма всех переносов TP) доступен для отчёта
- [ ] Smart-list «Replanned tracking points» предоставляет экспорт

### REQ-BOOK-021 — Replanning — API и чат с reason code

**Приоритет:** P1 | **Источник:** 2026 01 - Replan reason code & cancel (emma) - Google Slides.txt, Page 10


**Проверки:**
- [ ] API-маршрут replan принимает поля: replan_reason и comment (свободный текст)
- [ ] API не применяет проверку обязательности (mandatory/optional) согласно account-настройкам — обратная совместимость
- [ ] При replanning через API данные reason_code и comment логируются в чат так же, как при ручном replanning
- [ ] Лог replanning в чате видят все участники с доступом к чату (carrier и shipper)
- [ ] Роль инициатора (carrier/shipper/API user) фиксируется в записи replanning

### REQ-BOOK-022 — Admin — конфигурация причин отмены и переноса (Cancellation & Replan Management)

**Приоритет:** P1 | **Источник:** 2026 01 - Replan reason code & cancel (emma) - Google Slides.txt, Pages 4-5


**Проверки:**
- [ ] В карточке управления правами (admin rights management) есть секция Cancellation Management
- [ ] В той же карточке есть секция Replanning Management
- [ ] Каждая секция позволяет настроить: reason (mandatory/optional/do not request), dropdown-список значений, comments (mandatory/optional)
- [ ] Dropdown-список значений работает по той же логике что и metadata: если custom value disabled — только dropdown, если enabled — dropdown + свободный ввод
- [ ] При отсутствии custom-списка — используется свободный текстовый ввод
- [ ] Только пользователь с правами администратора аккаунта может изменить эти настройки

### REQ-BOOK-023 — PUT /shipments/contents — пересчёт Weight/Volume/CW/LM

**Приоритет:** P0 | **Источник:** 2024 09 - Update Packing List (TMS) - Google Slides.txt, Pages 6-8


**Проверки:**
- [ ] PUT /shipments/contents полностью заменяет текущий packing list новыми данными
- [ ] После PUT пересчитывается Volume (если переданы габариты), Weight, Linear Meters, Chargeable Weight
- [ ] Если размерные данные не переданы — Volume и LM очищаются (не сохраняются старые значения)
- [ ] TOTAL по всем позициям пересчитывается и обновляется синхронно
- [ ] Замена packing list логируется в чате: 'X обновил упаковочный лист, предыдущий список был: [предыдущее содержимое]'
- [ ] PUT корректно работает на уровнях: ACCOUNT, GALAXY-SHIPMENT, CARRIER-GALAXY
- [ ] Обновление через PUT триггерит соответствующий webhook

### REQ-BOOK-024 — POST /shipments/contents — пересчёт при добавлении позиции

**Проверки:**
- [ ] POST /shipments/contents добавляет новую позицию к существующему packing list
- [ ] После POST пересчитываются Volume, Weight, LM, CW с учётом всех позиций
- [ ] TOTAL обновляется после добавления
- [ ] Добавление позиции логируется в чате
- [ ] POST /shipments/contents триггерит webhook
- [ ] POST корректно обрабатывает позицию с опасными грузами (dangerous_goods с un_code, class_of_danger и т.д.)

### REQ-BOOK-025 — PATCH /shipments/contents — патч по content_id с пересчётом CW

**Приоритет:** P1 | **Источник:** 2024 09 - Update Packing List (TMS) - Google Slides.txt, Pages 13-14


**Проверки:**
- [ ] PATCH /shipments/contents/{id} обновляет конкретную позицию по её content_id
- [ ] После PATCH пересчитывается CW для изменённой позиции и TOTAL
- [ ] Если у позиции нет размеров после PATCH — CW очищается или пересчитывается на основе weight
- [ ] PATCH корректно возвращает content_id для каждой позиции (включая позиции с quantity > 1)
- [ ] Изменения через PATCH логируются в чате с указанием старых и новых значений
- [ ] PATCH недоступен для full truck позиций, где весовая система не поддерживает его применение

### REQ-BOOK-026 — Обновление packing list при подтверждении пикапа/доставки (TP confirmation)

**Приоритет:** P1 | **Источник:** 2024 09 - Update Packing List (TMS) - Google Slides.txt, Pages 2-5


**Проверки:**
- [ ] При подтверждении пикапа вручную пользователь может обновить фактический packing list
- [ ] При подтверждении доставки пользователь может обновить packing list выгрузки
- [ ] API-маршрут подтверждения TP принимает опциональный блок contents[] с полным описанием груза
- [ ] Volume пересчитывается из переданных габаритов если они предоставлены
