# Система уведомлений — Notifications & Business Log

> Источник требований: REQ-BOOK-031..036 | Слайды: 2025 09 - EPIC - Notification

---

## Обзор

Платформа Shiptify предоставляет многоуровневую систему уведомлений:

| Компонент | Описание |
|-----------|----------|
| **Business Log** | Журнал всех событий платформы с полным payload |
| **Notification Settings** | Подписка на события по типам, режимам транспорта и STY-кодам |
| **Email Digest** | Периодические сводки по событиям (spot / 1h / 2h / 4h / 8h) |
| **Vacation Mode** | Временное отключение с переадресацией |
| **Per-object Mute/Follow** | Переопределение настроек на уровне конкретного объекта |

---

## Business Log (Notification Center) — REQ-BOOK-031

Business Log — центральный журнал всех событий платформы, доступный через **новую вкладку в риббоне аккаунта**.

### Структура записи

| Поле | Описание |
|------|----------|
| **ERROR** | Флаг ошибки |
| **HOW** | Manual / API / Integration |
| **WHAT** | Тип объекта (Shipment, Booking, Slot...) |
| **WHY** | Тип действия (создание, обновление, отмена...) |
| **WHO** | Пользователь + email |
| **WHEN** | Дата и время события |
| **Object ID** | Кликабельная ссылка на объект |
| **Object name** | Название объекта |

### Поведение

- Клик на **Object ID** → открывает страницу объекта (если объект существует и у пользователя есть доступ)
- Клик на **строку** (не на ID) → открывает боковую модалку с деталями: payload, notified people, updated/previous values
- Записи хранятся **2 недели**, затем автоматически удаляются

### Фильтрация

Business Log поддерживает фильтры: users, states, object type, ID/name search, flow (manual/api), date range, быстрые периоды (last 1h / 2h / 4h).

### API-ошибки

API-ошибки попадают в Business Log с **полным payload запроса** и деталями ошибки — особенно важно для ADB/SADB потоков.

## Настройки уведомлений — REQ-BOOK-032

Пользователь настраивает уведомления в **профиле пользователя** на уровне каждого аккаунта отдельно.

### Параметры подписки

| Параметр | Варианты |
|----------|---------|
| **Тип события** | created / updated / cancelled / replanned и др. |
| **Тип объекта** | Shipment, Booking, Slot, Tracking Point... |
| **Канал** | In-app bell + Email |
| **Режим транспорта** | All или конкретные режимы (Air/Sea/Road) |
| **STY-коды** | Диапазон Tracking Point кодов для уведомлений |

### Правила

- Настройки **раздельны по ролям**: booker/creator, member of account, spectator
- Если событие подходит под две роли одновременно — отправляется **одно** уведомление
- Multi-account: настройки применяются на уровне каждого аккаунта отдельно

## Email Digest — REQ-BOOK-033

Пользователь выбирает **частоту дайджеста** для каждого типа событий:

| Режим | Описание |
|-------|----------|
| **spot** | Одно письмо на событие (немедленно) |
| **1h / 2h / 3h / 4h / 8h** | Сводка за период |

### Формат дайджеста

```
user(account) action element at date [SEE HERE]
```

- `SEE HERE` — кликабельная короткая ссылка на объект
- Письмо группируется по бизнес-объектам: BOOKING, TRACKING, SLOT
- Если за период не было событий — письмо **не отправляется**
- Cron-job работает по UTC-интервалам (не привязан к timezone клиента)
- Разные типы событий могут иметь **разные частоты** (например: created каждый 1h, packing list каждые 4h)

## Vacation Mode — REQ-BOOK-034

Режим отпуска позволяет временно приостановить все уведомления.

**Функции:**
- Включение vacation mode → все in-app и email уведомления прекращаются
- Опциональная **переадресация**: пользователь указывает коллегу, который получает уведомления на период отсутствия
- После выключения vacation mode уведомления возобновляются немедленно
- Business Log продолжает накапливать события (без пропуска записей)

> **Ограничение:** переадресация работает только для прямых уведомлений, не для digest.

## Уведомление о просроченном pickup/delivery — REQ-BOOK-035

Система автоматически генерирует уведомление в двух случаях:

| Условие | Текст уведомления |
|---------|------------------|
| Дата пикапа прошла, статус всё ещё `PLANNED` | `<Shipment Name> collect isn't confirmed yet but expected` |
| Дата доставки прошла, статус всё ещё `IN_TRANSIT` | `<Shipment Name> delivery isn't confirmed yet but expected` |

**Правила:**
- Пользователь подписывается на этот тип уведомлений **отдельно** в настройках
- Уведомление не дублируется, если уже было отправлено для данной отправки
- В UI отображается статус `Delivered (ESTIMATE)` при дате доставки > NOW без ручного подтверждения

## Mute / Follow на уровне объекта — REQ-BOOK-036

Позволяет переопределить глобальные настройки для конкретного объекта (отправки, бронирования, слота).

### Опции

| Действие | Результат |
|----------|-----------|
| **Don't notify me anymore for this object** | Заглушает все уведомления для данного объекта |
| **Notify me for this object** | Переопределяет глобальные настройки — включает уведомления |

**Правила:**
- Mute/Follow на уровне объекта имеет **приоритет** над глобальными настройками аккаунта
- Mute распространяется на **оба** канала: in-app и email
- В **чате** доступны команды `/mute` и `/follow` для управления уведомлениями чата объекта
- Список заглушённых объектов доступен в настройках пользователя
