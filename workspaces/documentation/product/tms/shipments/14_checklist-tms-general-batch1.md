## Раздел 1: Grouping 2.0 — Создание и управление группами

### REQ-GRP-001 — Создание группы из одного отправления
**Приоритет:** P1 | **Источник:** 00 - Ongoing Specs__2022 06 - Grouping 2.0.txt

**Проверки:**
- [ ] Кнопки «GROUP DEPARTURE» и «GROUP ARRIVAL» отображаются при выборе одного отправления (SH)
- [ ] При выборе 1 SH открывается тот же UX-поток создания группы, что и при выборе 2+ SH
- [ ] Опция создания манифеста доступна даже при группировке из 1 SH
- [ ] Grouping ID генерируется при создании группы из 1 SH
- [ ] Отправление с группой из 1 SH отображается в листинге с корректным Grouping ID
- [ ] Роль перевозчика не влияет на видимость кнопок группировки (доступно владельцу)

### REQ-GRP-002 — Группировка отправлений с уже существующей группой
**Приоритет:** P1 | **Источник:** 00 - Ongoing Specs__2022 06 - Grouping 2.0.txt

**Проверки:**
- [ ] При выборе набора SH, где часть уже сгруппирована, кнопки «GROUP DEPARTURE» и «CREATE MILKRUN» отображаются (не скрываются)
- [ ] Открывается модальное окно с вариантами: «Создать новую сгруппированную отправку», «Добавить к существующей группе», «Создать новый Milkrun», «Добавить к существующему Milkrun»
- [ ] При выборе «Добавить к существующей группе» используется тот же Grouping ID, а не генерируется новый
- [ ] При выборе «Создать новую сгруппированную отправку» генерируется новый Grouping ID
- [ ] После операции все задействованные SH корректно получают актуальный Grouping ID
- [ ] Предыдущий шаг «освобождения» (FREE) перед повторной группировкой больше не требуется

### REQ-GRP-003 — Освобождение (FREE) одного SH из группы двух
**Приоритет:** P1 | **Источник:** 00 - Ongoing Specs__2022 06 - Grouping 2.0.txt

**Проверки:**
- [ ] При освобождении 1 SH из группы из 2 оставшийся SH сохраняет Grouping ID и Manifest ID
- [ ] Если пользователь хочет удалить Grouping ID у оставшегося SH, он обязан выбрать оба SH и применить «FREE»
- [ ] После FREE одного SH из группы из 3 оставшиеся 2 SH сохраняют Grouping ID и Manifest ID (позитивный сценарий)
- [ ] Grouping ID отображается в интерфейсе даже если Manifest не был активирован
- [ ] Попытка FREE одиночного SH при неверных правах доступа блокируется с информативным сообщением
- [ ] История событий (audit log) фиксирует операцию FREE с указанием пользователя и временной метки

- [ ] Поле Grouping ID индексировано и доступно для поиска/фильтрации

### REQ-GRP-005 — Типы группировки: определения и правила
**Приоритет:** P0 | **Источник:** 00 - Ongoing Specs__2022 12 - MD, Docs vs. BK_SH vs. Grouping.txt

**Проверки:**
- [ ] «Grouped Departure» = 1 перевозчик, N SH, 1 точка погрузки, 1 точка доставки — создаётся корректно
- [ ] «Grouped Delivery» = 1 перевозчик, N SH, 1 точка погрузки, N точек доставки — создаётся корректно
- [ ] «Grouped Shipments» = 1 перевозчик, N SH, N точек погрузки, 1 точка доставки — создаётся корректно
- [ ] «Milkrun» = 1 перевозчик, N SH, N точек погрузки, N точек доставки — создаётся корректно
- [ ] «Mono Shipment» = 1 перевозчик, 1 SH, 1 точка погрузки, 1 точка доставки — не ошибочно классифицируется как группа
- [ ] «Multi Container» = 1 перевозчик, N SH, 1 точка погрузки, 1 точка доставки — разграничивается от Grouped Departure

### REQ-GRP-006 — Загрузка документов на сгруппированные отправления
**Приоритет:** P1 | **Источник:** 00 - Ongoing Specs__2022 12 - MD, Docs vs. BK_SH vs. Grouping.txt

**Проверки:**
- [ ] При загрузке документа на один SH из группы отображаются варианты: «This shipment», «All shipments», «Selection», «Same Ship / Same Destination / Same Pick-up»
- [ ] Выбор «All shipments» распространяет документ на все SH в группе
- [ ] Выбор «Selection» позволяет указать конкретные SH из группы
- [ ] Выбор «Same Ship» (для Multi Container / Sea Freight) корректно фильтрует SH по одному судну
- [ ] Выбор «Same Destination» (для Grouped Delivery) корректно фильтрует SH по пункту назначения
- [ ] Выбор «Same Pick-up» (для Grouped Shipments) корректно фильтрует SH по точке погрузки
- [ ] Документ, загруженный на все SH, доступен каждому получателю согласно правилам доступа

### REQ-GRP-007 — Уровни доступа к документам: Private / Limited / Public / Specific
**Приоритет:** P0 | **Источник:** 00 - Ongoing Specs__2022 12 - MD, Docs vs. BK_SH vs. Grouping.txt

**Проверки:**
- [ ] «Private» — только загрузивший аккаунт видит документ
- [ ] «Limited» — только Shipper и Carrier имеют доступ к документу
- [ ] «Public» — все аккаунты с доступом к SH (включая PML, Spectator) видят документ; новые участники, добавленные позже, также получают доступ автоматически
- [ ] «Specific» — можно вручную выбрать отдельных участников (Carrier, Shipper, Pick-up site, Delivery site, Spectator)
- [ ] Уведомления в чате о загрузке документа отправляются только аккаунтам, у которых есть доступ к документу
- [ ] В листинге документов отображается тег (Private / Limited / Specific), количество уведомлённых пользователей и детали при наведении

---

### REQ-GRP-008 — Уведомления о документах в сгруппированных отправлениях
**Приоритет:** P2 | **Источник:** 00 - Ongoing Specs__2022 12 - MD, Docs vs. BK_SH vs. Grouping.txt

**Проверки:**
- [ ] Уведомления о загрузке документа отправляются только аккаунтам, активированным для данного документа
- [ ] Тег с количеством («12») в листинге документов заменяет полный список уведомлённых пользователей
- [ ] Наведение на тег показывает список аккаунтов с доступом и список уведомлённых пользователей
- [ ] При удалении аккаунта из списка участников SH документ типа Public остаётся доступным для оставшихся
- [ ] Вкладка «Notifications» в настройках профиля позволяет управлять уведомлениями по документам
- [ ] Перевозчик может настроить уведомления по конкретным клиентам (Customer-level filtering) в своём профиле

## Раздел 2: Freight Units (FU) — Управление грузовыми единицами

### REQ-FU-001 — Создание Freight Unit через API (VPE → STY)
**Приоритет:** P0 | **Источник:** 00 - Ongoing Specs__2022 03 - VPE Freight Unit Generation.txt

**Проверки:**
- [ ] API FU-100: Veepee может создать FU в системе Shiptify; FU получает корректные поля: FU_Veepee_ID, FU_Name, аккаунт Shipper, Pick up location, Final destination, due dates, массив грузов
- [ ] API FU-101: Veepee может обновить существующий FU; данные перезаписываются корректно
- [ ] API FU-102: Veepee может удалить FU, но только если FU не назначен ни на одну транспортировку
- [ ] Попытка удалить FU, назначенный на транспортировку, возвращает ошибку с корректным сообщением
- [ ] FU содержит массив: тип груза, количество, свойства груза (габариты, вес, специфика, признак опасного груза)
- [ ] При успешном создании FU система возвращает FU_Shiptify_ID

### REQ-FU-002 — Веб-хуки от Shiptify к Veepee
**Приоритет:** P0 | **Источник:** 00 - Ongoing Specs__2022 03 - VPE Freight Unit Generation.txt

**Проверки:**
- [ ] FU-200: При создании SH с FU Shiptify отправляет веб-хук с SH ID и массивом FU_ID

- [ ] Veepee может получить детали SH по API-900 (GET SH details)

### REQ-FU-003 — Статусы Freight Unit
**Приоритет:** P0 | **Источник:** 00 - Ongoing Specs__2022 03 - VPE Freight Unit Generation.txt

**Проверки:**
- [ ] «Pending Collect» — FU не назначен ни на одно SH/BK; отображается оранжевым
- [ ] «Assigned» — FU назначен на 1+ SH/BK; отображается синим если дата погрузки ≤ сегодня
- [ ] «In Transit» — FU полностью назначен до пункта назначения; промежуточный статус
- [ ] «Pending Dispatch» — FU назначен до Dispatch Hub, но не до конечного пункта
- [ ] «Delivered» — FU доставлен в конечный пункт назначения
- [ ] Переходы статусов строго соответствуют бизнес-логике; некорректные переходы блокируются
- [ ] Цвета в листинге соответствуют статусам: оранжевый — нет SH, серый — есть BK (не отменён), синий — есть SH и дата погрузки ≤ сейчас

### REQ-FU-004 — Поля Last Transit Location и маршрутные таблицы
**Приоритет:** P1 | **Источник:** 00 - Ongoing Specs__2022 03 - VPE Freight Unit Generation.txt

**Проверки:**
- [ ] Last Transit Location автоматически обновляется при назначении FU на SH (используется адрес доставки последнего SH в цепочке)
- [ ] Last Transit Location date/hour обновляется при подтверждении доставки последнего SH

- [ ] Таблица DISPATCH работает аналогично COLLECT с полями Dispatch Hub

## Раздел 4: Auth / SSO — Аутентификация и единый вход

### REQ-AUTH-001 — Требования к аутентификации через Identity Provider
**Приоритет:** P0 | **Источник:** drive-download__SAML__02. User Guide - Authentification & SSO-v1.0.txt

**Проверки:**
- [ ] Все приложения используют Chanel IDP (AD) для аутентификации; локальная аутентификация запрещена (кроме настройки и поддержки)
- [ ] SSO реализован для конечных пользователей независимо от места хостинга приложения
- [ ] После успешной аутентификации приложение проверяет права доступа пользователя (авторизация отделена от аутентификации)
- [ ] IDP передаёт атрибуты AD (login, country, division, role, location, group membership) для управления правами
- [ ] Приложение не хранит локальные пароли пользователей; все сессии управляются через IDP-токены
- [ ] Self-Service Password Reset (SSPR) доступен пользователям и работает корректно

---

### REQ-AUTH-002 — SAML 2.0 аутентификация
**Приоритет:** P0 | **Источник:** drive-download__SAML__02. User Guide - Authentification & SSO-v1.0.txt

**Проверки:**
- [ ] SP-сертификат генерируется корректно (максимальный срок действия 3 года); элемент AuthnContextClassRef удалён из SAML-запросов
- [ ] SP Metadata генерируется и принимается IDP при конфигурации
- [ ] SAML-поток: IDP разбирает запрос → запрашивает login/password при необходимости → возвращает закодированный SAML-ответ → приложение верифицирует ответ → пользователь авторизован
- [ ] Алгоритм хэширования SHA-256 применяется обязательно
- [ ] Claim Rules (Name Identifier, email, IGI и др.) передаются в SAML-токене корректно
- [ ] При истечении или компрометации SP-сертификата обновление происходит через соответствующий процесс без прерывания сервиса

---

### REQ-AUTH-003 — OAuth 2.0 / OpenID Connect — Implicit Grant (SPA)
**Приоритет:** P0 | **Источник:** drive-download__SAML__02. User Guide - Authentification & SSO-v1.0.txt

**Проверки:**
- [ ] SPA использует Implicit Grant; поток выполняется через дефолтный браузер устройства
- [ ] После аутентификации IDP возвращает ID Token и Access Token во фрагменте URL
- [ ] SPA использует ID Token для получения атрибутов пользователя, Access Token — для вызова API
- [ ] Client Secret не хранится в SPA (Implicit Grant использует public client)
- [ ] Embedded WebView запрещён; используется только внешний user agent (браузер)
- [ ] Endpoints IDP: Authorization, Token, UserInfo — доступны и корректно отвечают

### REQ-AUTH-004 — OAuth 2.0 Authorization Code Grant (Web App / Native App)
**Приоритет:** P0 | **Источник:** drive-download__SAML__02. User Guide - Authentification & SSO-v1.0.txt

**Проверки:**
- [ ] Web App инициирует поток: редирект → IDP → Authorization Code → обмен Code на ID Token + Access Token с использованием Client Id и Client Secret
- [ ] Pre-shared Key (Client Secret) имеет срок действия 3 года и обновляется через соответствующий процесс
- [ ] Native (iOS) App использует Authorization Code Grant с PKCE; поток выполняется через внешний браузер
- [ ] Access Token используется для вызовов Web API; ID Token — для получения атрибутов пользователя
- [ ] Подпись JWT-токена верифицируется клиентским приложением и resource server через OpenID Connect discovery endpoint
- [ ] При необходимости доступа к O365 ресурсам используется Azure AD вместо ADFS

### REQ-AUTH-005 — OAuth 2.0 Client Credentials Grant (M2M / Daemon)
**Приоритет:** P1 | **Источник:** drive-download__SAML__02. User Guide - Authentification & SSO-v1.0.txt

**Проверки:**
- [ ] Daemon-приложение аутентифицируется через Client Id и Client Secret без участия конечного пользователя
- [ ] Pre-shared Key (Client Secret) обновляется каждые 3 года
- [ ] Access Token получается напрямую от IDP и используется для вызовов Web API
- [ ] APIM gateway верифицирует подпись токена; Web API не обязан повторно проверять валидность токена (если соединение APIM→Web API защищено)
- [ ] Web API защищён одним из методов: HTTP Basic Auth, SSL Mutual Auth, IP Whitelisting, Azure VNet
- [ ] Логирование вызовов (call metadata) для аналитических целей работает через APIM

### REQ-AUTH-006 — TLS аутентификация (M2M с сертификатами)
**Приоритет:** P1 | **Источник:** drive-download__SAML__02. User Guide - Authentification & SSO-v1.0.txt

**Проверки:**
- [ ] TLS 1.2 используется для защиты M2M коммуникаций (обязательно)
- [ ] CSR генерируется на устройстве, где запущено приложение; длина ключа RSA ≥ 2048 бит; приватный ключ защищён passphrase и никогда не передаётся
- [ ] Subject Alt Name (SAN) содержит FQDN приложения; Organization, OU, Location, Email заполнены
- [ ] TLS handshake: client hello → server hello + certificate request → client key exchange + client certificate → взаимная верификация → encrypted session
- [ ] При компрометации или истечении клиентского сертификата создаётся новый CSR с новой ключевой парой
- [ ] APIM gateway проверяет клиентский сертификат; Web API доверяет соединению через APIM
