# Карта продукта — Типы пользователей и сценарии использования

## Типы пользователей (User Types)

### 1. Shipper (Грузоотправитель)

**Кто это:** Компания, которая отправляет грузы. Основной покупатель услуги.

**Что умеет:**
- Создавать заявки на перевозку (SR и QR)
- Следить за статусом своих перевозок
- Управлять слотами на своих складах
- Загружать и запрашивать документы
- Настраивать локации, партнёров, пользователей
- Экспортировать данные в Excel
- Создавать шаблоны перевозок (Shipment Templates)
- Управлять заказами (Orders)

**Роли внутри Shipper-аккаунта:**
| Роль | Права |
|------|-------|
| `api` | Стандартный доступ (создание, просмотр, управление) |
| `self_admin` | Управление пользователями аккаунта |
| `spectator` | Только просмотр перевозок |
| `invoicing` | Доступ к инвойсингу |

---

### 2. Carrier (Перевозчик)

**Кто это:** Транспортная компания, которая выполняет перевозки.

**Что умеет:**
- Получать и подтверждать/отклонять заявки (SR)
- Отвечать на запросы котировок (QR)
- Обновлять статусы трекинга (подтверждать TrackingPoints)
- Бронировать слоты на складах shipper'а
- Выставлять счета (Pre-Invoice, Invoice)
- Загружать документы (CMR, инвойсы)
- Управлять своими водителями (для Driver App)

**Роли внутри Carrier-аккаунта:**
| Роль | Права |
|------|-------|
| `api` | Стандартный доступ |
| `self_admin` | Управление пользователями |
| `tracking` | Только обновление трекинга |

---

### 3. Admin (Администратор Shiptify)

**Кто это:** Внутренний сотрудник Shiptify с полным доступом.

**Что умеет:**
- Полный доступ к любому аккаунту
- Управление мастер-данными (типы груза, payment terms, currencies)
- Управление настройками аккаунтов
- Создание/удаление аккаунтов
- Принудительная смена статусов
- Реактивация отменённых перевозок
- Доступ к Back-office

---

### 4. Account Manager (Внутренний менеджер Shiptify)

**Кто это:** Менеджер по работе с клиентами в back-office системе.

**Что умеет:**
- Управление Sales Accounts (CRM)
- Создание и ведение Opportunities (пайплайн продаж)
- Touchpoint менеджмент (@mention уведомления)
- Управление Galaxies (группы аккаунтов)
- Настройка Billing (MRR, billing accounts)
- Онбординг новых клиентов

**Работает через:** `workspaces/back-office`

---

### 5. Driver (Водитель)

**Кто это:** Водитель, выполняющий перевозку. Работает через мобильное приложение.

**Что умеет:**
- Просматривать назначенный рейс
- Подтверждать pick-up (забор груза)
- Подтверждать delivery (доставку)
- Уведомлять о задержке
- Фотографировать CMR-документ

**Работает через:** `workspaces/mini-apps` → `driver-app`
**Языки:** 17 языков (мультиязычный)

---

### 6. Operator / Dock Manager (Оператор склада)

**Кто это:** Сотрудник склада, управляющий воротами и слотами.

**Что умеет:**
- Подтверждать/отклонять запросы на слоты
- Переназначать ворота (dock doors)
- Управлять расписанием в Slotify
- Видеть нагрузку на склад

---

### 7. Spectator (Наблюдатель)

**Кто это:** Внешний участник (клиент shipper'а, таможенный брокер).

**Что умеет:** Только просмотр перевозок, которыми с ним поделились.

## Основные сценарии (User Journeys)

### Journey 1: Стандартная перевозка (SR)

```
[Shipper]
  1. Создаёт заявку в CSW wizard
  2. Выбирает одного перевозчика
  3. Отправляет заявку → SR создан

[Carrier]
  4. Получает email с заявкой
  5. Подтверждает бронирование

[Система]
  6. Shipment автоматически создаётся

[Carrier]
  7. Забирает груз → подтверждает TrackingPoint "departure"
  8. Доставляет груз → подтверждает TrackingPoint "arrival"

[Shipper]
  9. Видит статус "delivered"
  10. Подтверждает POD (Proof of Delivery)

[Carrier]
  11. Выставляет Pre-Invoice
  12. Shipper верифицирует → Invoice
```

---

### Journey 2: Тендер (QR — Quote Request)

```
[Shipper]
  1. Создаёт QR, выбирает 3+ перевозчиков
  2. Устанавливает reply_before deadline

[Carrier A, B, C]
  3. Получают запрос котировки
  4. Отвечают с ценой и условиями

[Shipper]
  5. Сравнивает котировки
  6. Принимает лучшую (Carrier B)

[Система]
  7. Carrier B → Shipment создан
  8. Carrier A, C → получают уведомление об отклонении

[Дальше как Journey 1: трекинг → инвойс]
```

---

### Journey 3: Milkrun (несколько отправлений, один маршрут)

```
[Shipper]
  1. Создаёт заявку с несколькими pre-shipments
  2. Второй pre-shipment совпадает по локации или дате с первым
  → Система определяет как Milkrun

[Или через Grouping Page]
  1. Shipper нажимает [MILKRUN] на странице группировки
  2. Выбирает существующие Shipment'ы для объединения

[Carrier]
  3. Получает один маршрут с несколькими точками
  4. Обновляет трекинг по каждой точке маршрута
```

## Граничные случаи (Corner Cases)

### Заявки (Requests)

| Ситуация | Что происходит |
|---------|---------------|
| QR: ни один перевозчик не ответил до deadline | QR истекает, Shipper получает уведомление, создаёт новый QR или SR |
| Carrier отклонил SR | `ShipmentRequest.status = cancelled`, Shipper получает email, может создать новую заявку |
| Carrier подтвердил, потом хочет отказаться | Требует отмены Shipment (уже создан) с уведомлением Shipper |
| Shipper редактирует заявку после отправки | Только при статусе `pending`, пока Carrier не подтвердил |
| Дубликат заявки (аварийная отправка) | Функция "Дублировать" создаёт новый SR с теми же данными |

### Трекинг (Tracking)

| Ситуация | Что происходит |
|---------|---------------|
| Carrier не подтвердил pick-up вовремя | `TrackingPoint.status = delayed`, Shipper видит предупреждение |
| Два источника трекинга одновременно (Driver App + P44) | Последнее обновление побеждает, оба логируются |
| Перевозка отменена после частичного трекинга | `Shipment.status = canceled`, уже подтверждённые TP сохраняются в истории |
| POD загружен, но Shipper отклонил | `pod_status = declined`, Carrier должен загрузить новый |
| Задержка + инцидент | TrackingPoint помечается incident, Shipper может создать Claim |

### Слоты (Slots)

| Ситуация | Что происходит |
|---------|---------------|
| Два Carrier'а хотят один и тот же слот | Первый подтверждённый получает слот, второй должен выбрать другое время |
| Слот забронирован, Shipment отменён | Слот автоматически освобождается |
| Carrier приехал не в своё окно | Оператор может переназначить слот вручную |
| Склад закрыт (праздник) | Слоты на этот день недоступны (Location Calendar) |

### Мультиарендность (Galaxy)

| Ситуация | Что происходит |
|---------|---------------|
| Shipper входит в несколько аккаунтов (Galaxy) | Видит данные всех аккаунтов своей Galaxy |
| Managed Account — Shipper не может сам бронировать | Все заявки идут через управляющий аккаунт |
| Division-based ACL: user видит только свой division | ShipperACL ограничивает данные по `division_id` |
