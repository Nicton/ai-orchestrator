# Corner Cases — Граничные случаи

Что происходит, когда всё идёт не по плану. Каждый случай описывает ситуацию, поведение системы и код, который это обрабатывает.

---

## Заявки (Requests)

| Ситуация | Поведение системы | Код |
|---------|------------------|-----|
| QR: ни один Carrier не ответил до deadline | QR истекает. Shipper получает email. Может создать новый QR или SR с теми же данными | `app/cron/` → задача очистки QR + `mailQrExpiredToShipper` |
| Carrier отклонил SR | `ShipmentRequest.status = cancelled`. Shipper получает email. Может создать дубликат заявки | `notifyShipmentRequestCancelled()` |
| Carrier подтвердил SR, потом хочет отказаться | Требует отмены уже созданного Shipment. Shipper должен согласиться | Отмена Shipment → `runCancelShipmentIntegrations()` |
| Shipper хочет изменить заявку после отправки | Редактирование доступно только пока `status = pending`. После подтверждения — только отмена и повторное создание | Frontend: кнопка "Edit" скрыта при `status ≠ pending` |
| Два Shipper'а бронируют одного и того же Carrier на одно время | Carrier получает оба запроса, сам решает — какой принять | Нет автоматической блокировки конфликтов |
| Carrier не имеет доступа к интеграции (нет API) | SR/QR создаётся как обычно. Интеграция пропускается. Трекинг — ручной | Интеграция опциональна, не блокирует флоу |

## Трекинг (Tracking)

| Ситуация | Поведение системы | Код |
|---------|------------------|-----|
| Carrier не подтвердил pick-up вовремя | `TrackingPoint.status = delayed`. Shipper видит предупреждение на странице Shipment. Может отправить напоминание | `calculateShipmentStatus()` + `remindTracking()` |
| Driver App и UI обновляют TP одновременно | Последнее обновление побеждает. Оба источника логируются | `TrackingPoint.updated_at` |
| Перевозка отменена после частичного трекинга | `Shipment.status = canceled`. Уже подтверждённые TP сохраняются в истории | `runCancelShipmentIntegrations()` |
| POD загружен, но Shipper отклонил | `pod_status = declined`. Carrier должен загрузить новый POD. Shipper получает уведомление о повторном запросе | `requestProofOfDelivery()` |
| Задержка + инцидент | TP помечается как `incident`. Shipper может создать Claim прямо из TP | `TrackingPoint.status = incident` → кнопка "Create Claim" |
| P44/Shippeo перестали отвечать | Автоматический трекинг прекращается. Carrier переходит на ручное обновление. Нет алерта для пользователя | Cron ловит ошибки в лог, не падает |
| Два разных источника дают разный статус | Ручное обновление через UI имеет приоритет над автоматическим | `TrackingPoint.source` field |

## Слоты (Slots)

| Ситуация | Поведение системы | Код |
|---------|------------------|-----|
| Два Carrier'а хотят один и тот же слот | Первый подтверждённый получает слот. Второй должен выбрать другое время | `app/services/slots/` — проверка конфликтов |
| Слот забронирован, но Shipment отменён | Слот автоматически освобождается (SlotShipment удаляется) | Trigger при `Shipment.status = canceled` |
| Carrier приехал не в своё временное окно | Оператор может переназначить слот вручную в Slotify | UI: drag-and-drop в Slotify |
| Склад закрыт в праздничный день | Слоты на этот день недоступны в booking calendar | `Location.calendar` → выходные/праздники |
| Слот подтверждён, но Carrier не приехал | Слот помечается как "no-show" (опционально). Следующий интервал свободен | Ручное действие оператора |
| Warehouse capacity exceeded | Система предупреждает, но не блокирует. Оператор решает | UI: цвет ячейки меняется на красный |

## Инвойсинг (Invoicing)

| Ситуация | Поведение системы | Код |
|---------|------------------|-----|
| Carrier выставил сумму выше котировки | Shipper видит delta (разницу). Может оспорить Pre-Invoice | `PreInvoice.status = disputed` |
| Shipper не верифицирует инвойс долго | Напоминание через email (cron) | `app/cron/` → группа "Инвойсы" |
| Курс валюты изменился за время перевозки | Система хранит курс на дату создания инвойса | `Currency.rate` зафиксирован при создании |
| SAP интеграция недоступна | Invoice создаётся локально. Синхронизация в SAP выполнится при следующем cron-запуске | Retry через cron |

## Мультиарендность (Galaxy / Multi-Account)

| Ситуация | Поведение системы | Код |
|---------|------------------|-----|
| Shipper входит в несколько Galaxy | Видит данные всех аккаунтов своей Galaxy | `Galaxy` + `GalaxyAccount` |
| Managed Account — Shipper не может сам бронировать | Все заявки идут через управляющий аккаунт | ACL: `managed = true` |
| Пользователь потерял доступ к division | Данные этого division становятся невидимы. Уже созданные записи сохраняются | `ShipperACL.division_id` |
| Один пользователь = несколько аккаунтов | Пользователь переключается между аккаунтами через UI. Данные изолированы | `account_id` в каждом запросе |

## Интеграции с перевозчиками

| Ситуация | Поведение системы | Код |
|---------|------------------|-----|
| API перевозчика недоступен | Job уходит в очередь с retry. После N попыток — `failed`. Уведомление в Telegram | `retryAttempts` + Kue failed state |
| API перевозчика вернул ошибку (неверные данные) | Job помечается как `failed`. Carrier получает уведомление. Требует ручного исправления данных | Специфичные handler'ы по перевозчику |
| Дубликат бронирования в API перевозчика | Интеграция проверяет наличие уже существующего booking по tracking_number | `findTrackingPointByExternalKey()` |
| Перевозчик изменил API (версия обновилась) | Требует обновления `provider.js`. До обновления — автоматическое бронирование недоступно, ручной режим | `app/services/integration/[carrier]/provider.js` |
