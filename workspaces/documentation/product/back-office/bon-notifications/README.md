# BON (BO Notifications) — Система уведомлений Back-Office

## Статус

**Система BON находится в стадии разработки.** В кодовой базе на данный момент присутствует только старая модель `MsgNotifUser` (предпочтения уведомлений для операционных модулей). Полная система BON (inbox-интерфейс, таблица `bo_notifications`, парсинг @-упоминаний в тачпоинтах, email-доставка) запланирована к реализации.

## Назначение BON

BON — это inbox-система уведомлений для внутренней команды Shiptify. Основной триггер: когда AM упоминает коллегу через `@имя` в тексте тачпоинта Sales Account, система создаёт уведомление и доставляет его получателю по email и в in-app инбокс.

**Ценность:** члены команды получают своевременные, действенные уведомления при упоминании в тачпоинтах Sales Account, с возможностью перепланировать или архивировать их.

## Структура объекта уведомления (планируемая схема)

| Поле | Тип | Описание |
|---|---|---|
| `source_l1` | String | Категория источника верхнего уровня (например, Sales Account) |
| `source_l2` | String | Подкатегория (например, Touchpoint) |
| `enriched_text` | Text | Текст уведомления с контекстом |
| `sender_id` | FK → users | Кто создал уведомление |
| `recipient_id` | FK → users | Кто получает уведомление |
| `status` | Enum | `pending` / `archived` / `replanned` |
| `read_status` | Boolean | Прочитано/непрочитано |
| `planned_at` | DateTime | Дата перепланирования (для статуса `replanned`) |

## Вкладки Inbox

| Вкладка | Описание |
|---|---|
| **Pending** | Непрочитанные уведомления |
| **To Do — This Week** | Перепланированные на эту неделю |
| **To Do — Next Week** | Перепланированные на следующую неделю |
| **To Do — Later** | Перепланированные на более далёкую дату |
| **Archived** | Архив (автоматическое архивирование через 20 дней) |

## Модальное окно Plan Task

Быстрые варианты при перепланировании уведомления:

| Вариант | Время |
|---|---|
| This afternoon | 14:00 текущего дня |
| Tomorrow | 07:00 следующего дня |
| In 2 Days | 07:00 через 2 дня |

## Клавиатурные сокращения

| Клавиша | Действие |
|---|---|
| `ENTER` | Архивировать уведомление |
| `A` | Архивировать |
| `CTRL+R` | Перепланировать |
| `R` | Ответить |
| `D` | Перепланировать (Replan) |

## Поток уведомлений (планируемый)

1. AM создаёт тачпоинт в Sales Account с `@упоминанием` коллеги
2. Backend парсит `@упоминания` из поля `notes` / `detailed_notes`
3. Для каждого упомянутого пользователя создаётся запись в `bo_notifications` (sender, recipient, source_l1=Sales Account, source_l2=Touchpoint)
4. Отправляется email получателю
5. Получатель видит уведомление во вкладке Pending инбокса
6. Получатель может: архивировать (ENTER/A), перепланировать (D/CTRL+R с modal), ответить (R)

## Текущее состояние в коде

Модель `MsgNotifUser` (`msg_notif_users`) — это **старая** система предпочтений уведомлений для операционных модулей:

| Поле | Описание |
|---|---|
| `thread_id` | Поток уведомлений |
| `user_id` | Пользователь |
| `is_quotes` | Уведомления о котировках |
| `is_tracking` | Уведомления о трекинге |
| `is_invoicing` | Уведомления об инвойсах |
| `is_claims` | Уведомления о претензиях |

Сервис `server/services/msg_notif_users.js` экспортирует только `destroyMsgNotificationUsers` (массовое удаление по user_id). Новая BON-система (`bo_notifications` таблица, inbox UI, email delivery) ещё не реализована в коде.

## Техническое ограничение

Система BON должна использовать существующий стек: Express.js + React 16 + Redux + Sequelize. Email-доставка — только внутренняя, без требований к дизайну (простой текст или минимальный HTML).
