#### 1. Integration Ownership + RACI
**Гипотеза:** Если назначить явных владельцев интеграционных зон и определить RACI — разработчики перестанут быть default «крайними» за внешние/бизнесовые неопределённости.

**Симптомы из интервью:**  
- «Нет ответственного за интеграции — разработчиков делают крайними»  
- «Нужна прослойка между разработчиками и французской стороной, нужен PM/ответственный»  
- «Поддержка DHL отвечает медленно, некому эскалировать»

**Риск без действия:** Blockers будут решаться медленно, разработчики тратят время на эскалации вместо engineering.

**Ожидаемый эффект:** Чёткая зона ответственности; меньше «футбола»; меньше конфликтов Dev ↔ QA ↔ external.

**Реализация:**  
Создать карту: Integration/Area → Owner → Backup → External contact → Escalation path → Docs link  
Начать с top-5 болезненных интеграций.

**Owner:** Team Lead + PO/PM  
**Срок:** MVP 1 неделя  
**KPI:** Time-to-owner; время закрытия blocker; % вопросов закрытых без эскалации  
**Done когда:** У каждой priority-интеграции есть owner, backup, путь эскалации и ссылка на документацию.

#### 2. Reject / Accept Policy + корректность метрик
**Гипотеза:** Если определить, что считается reject/accept и починить сбор метрик — метрики будут поддерживать улучшение, а не создавать страх.

**Симптомы из интервью:**  
- «Метрики собираются некорректно, reject-метрики тоже»  
- «Боимся реджектить, чтобы не испортить кому-то работу»  
- «Из-за этого неудобно работать»  
- «Евгений: не хочу, чтобы меня уволили из-за 10 реджектов»  
- «Команда: ставь любую оценку — всё равно ни на что не влияет»

**Риск без действия:** Команда будет скрывать проблемы; доверие к метрикам снизится; качество обсуждений станет эмоциональным.

**Ожидаемый эффект:** Прозрачный triage; меньше конфликтов; безопаснее фиксировать реальные проблемы.

**Реализация:**  
1-page policy: определение reject, категории, когда reject валиден/невалиден, review process, safe-guard без санкций в пилотном окне.  
Проверить текущий сбор метрик и починить.

**Owner:** QA Owner + Team Lead + PM  
**Срок:** Policy 1 неделя; validation 2-3 недели  
**KPI:** % rejects с категорией; disputed rejects; число конфликтов по reject  
**Done когда:** Reject definition утверждён; сбор проверен; спорные кейсы имеют review flow.

#### 3. As-is документация по интеграциям
**Гипотеза:** Если описать текущее поведение интеграций на основе кода, API, логов — onboarding и debugging станут быстрее.

**Симптомы из интервью:**  
- «На интеграциях нет документации»  
- «Логика лежит в сторонней API»  
- «Какие-то интеграции вообще не тестируются»  
- «Тестировщики не знают, что там происходит»  
- «Разработка и тестирование идут вслепую»  
- «Есть коэффициенты — интерфейс с логами, мониторинг, но это не документация»  
- Проблемные: teliae, все edifact-интеграции  
- «У Carrier хорошая документация» (есть эталон)

**Риск без действия:** Bus factor растёт; баги диагностируются медленнее; знания становятся «секретными».

**Ожидаемый эффект:** Быстрее root-cause; меньше повторных вопросов; выше качество onboarding.

**Реализация:**  
Начать с 2 проблемных интеграций: sequence diagram, endpoints, auth, mapping, configs, logs, типовые ошибки, тестовые данные.  
Шаблон: Integration Spec v1 (скопировать у Carrier как образец).  
Правило: нет обновления доки → нет Done.

**Owner:** Integration Owner + selected Dev + QA  
**Срок:** MVP 2 недели  
**KPI:** Время поиска ответа; число повторных вопросов; % docs с owner/review date  
**Done когда:** Минимум 2 интеграции имеют reviewed docs, owner, backup и last reviewed date.

#### 4. Requirements DoR + канал уточнений
**Гипотеза:** Если ввести минимальный Definition of Ready и единый канал уточнений — разработка будет получать более actionable задачи.

**Симптомы из интервью:**  
- «Нет аналитика, переводящего бизнес в разработку»  
- «Задачи абстрактные — надо идти напрямую к заказчику»  
- «Ramein в Jira не отвечает»  
- «Задача описана красиво, но непонятно, что делать»  
- «Мелкие правки тянут за собой большие изменения — нет аналитики»  
- «Нет доступа к сценариям тестирования»  
- «Спеки приходят недоделанные, недоработанные»  
- «Комментарии из задач превращаются в прострелы»

**Риск без действия:** Разработка стартует с «пустых» требований; rework и clarification loops продолжаются; приоритеты будут обходить процесс.

**Ожидаемый эффект:** Лучшее качество задач; меньше уточнений после grooming; выше предсказуемость delivery.

**Реализация:**  
DoR template: business goal, context, AC, affected modules/integrations, constraints, dependencies, contact, how to verify, open questions.  
Начать не с жёсткого давления, а с pilot на одном stakeholder path (Ivan → Ramein). Показать выигрыш.

**Owner:** PM/PO + Team Lead + Aleh  
**Срок:** 1-2 недели  
**KPI:** % tasks passing DoR; clarification comments after grooming; rework due to missed requirements  
**Done когда:** Пилотные задачи содержат цель, AC, ограничения, контакт и способ проверки.

#### 5. Minimal Source of Truth (SoT) / Knowledge Base
**Гипотеза:** Если создать и поддерживать минимальный SoT (карта системы + ключевые слои + интеграции + формулы/правила + окружения) — снизим время на разбор «как должно работать».

**Ожидаемый эффект:** Быстрее анализ задач/дефектов; меньше итераций «уточнить/переделать/перетестить»; выше предсказуемость релизов.

**Done когда:** Есть system map + ≥5 слоёв + integrations + environments + formulas/rules; команда ссылается на SoT в задачах/багах.

#### 9. Integration Readiness Checklist (gate перед стартом и релизом)
