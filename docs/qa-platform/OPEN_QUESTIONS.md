# QA Platform — вопросы к утру (ответишь — продолжу)

Реализовано ночью как раздел `/qa` внутри Searchify (API-first), задеплоено и
проверено E2E. Ниже — решения, которые я принял сам, и вопросы, где нужен твой выбор.

## Принял сам (можно поменять)
- **Стек:** не .NET из ТЗ, а стек Searchify (Fastify + Prisma + PostgreSQL + статический SPA), т.к. ты просил «внутри Searchify». Перенос в отдельное приложение/.NET позже = порт схемы и API.
- **Namespace:** всё под `/api/qa/*`, UI `/qa`, модели `Qa*`, чтобы не пересекаться с существующим.
- **Auth для агента/CI:** один `QA_API_TOKEN` (заголовок `x-qa-token`) + сессия Searchify для UI. Полноценные роли §21 не делал.
- **Coverage** считает сервер из связей+шагов+последних прогонов (не доверяю payload). Versioning — снэпшоты без diff-viewer.

## Вопросы
1. **Выносить ли в отдельное приложение** (и когда)? Если да — оставляем стек Node/Prisma или переписываем на .NET, как в ТЗ?
2. **Роли и пер-проектные API-токены** (Admin/QA Lead/Engineer/Automation/Viewer/API Agent, §21) — нужны сейчас или позже? Пока один общий токен.
3. **Связь с реальными данными Shiptify:** импортировать Requirements из Jira/Confluence/презентаций (у Searchify уже есть доступы) с автозаполнением Sources? Связать QA-проект с реальными репозиториями?
4. **Автотесты:** кто их гоняет (Playwright у Дмитрия Була?) — подключаем `tms-playwright-reporter`/`testcafe-reporter-tms` к реальному прогону? Reporter-пакеты публиковать в приватный `@shiptify` npm или ставить из пути?
5. **Grafana:** нужен готовый dashboard JSON поверх `/api/qa/projects/{id}/metrics`, или хватит встроенного дашборда в `/qa`?
6. **Diff-viewer версий** — нужен визуальный diff (сейчас снэпшоты + restore, без diff)?
7. **Приоритет доработок UI:** drill-down в traceability, экран версий, фильтры/Select-All в test-plan по фильтрам, step-level линковка автотеста к шагам через UI, секции-деревья (drag-n-drop). Что первым?
8. **Demo-проект «E2E Demo»** в проде — оставить как песочницу или удалить?

## Что точно доделать (бэклог, начну с твоих приоритетов)
- Requirement sources / acceptance criteria — добавление через UI (API уже есть).
- Версии (список/просмотр/restore) через UI.
- Test sections drag-n-drop + move endpoint в UI; вложенность визуально.
- Step-level coverage через UI (линковка AT → конкретные шаги).
- Импорт реальных требований/автотестов; seed-данные.
- Grafana dashboard JSON; пер-проектные токены/роли.
