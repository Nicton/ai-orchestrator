# ТЗ: модуль «Release Notes» в Searchify

> Для треда, реализующего фичу **внутри searchify** (`ai-orchestrator` app). Движок-методология уже
> описан в скиле `skills/31-release-notes-writer/SKILL.md` — этот документ задаёт обвязку (UI / API /
> хранение / перевод). Источник требований — постановка владельца (см. «Контекст» ниже).

## 1. Цель

По нажатию кнопки генерировать **release notes для трёх ЦА**, сохранять их в историю с шарабельной
ссылкой, давать удобное копирование (с подстановкой ссылки на отчёт в searchify) и **перевод по запросу**
(один отчёт читается на разных языках, перевод генерится только когда его попросили).

Генерация контента (анализ дельты, код-импакт, тексты отчётов) выполняется **движком-скилом**
`31-release-notes-writer` через LLM-пайплайн (`runRolePrompt`, Claude CLI — как у остальных ответов
searchify). Searchify отвечает за триггер, оркестрацию, хранение, ссылки, копирование, перевод.

## 2. Три отчёта (ЦА)

Скил выдаёт три самостоятельных markdown-отчёта на один релиз. Searchify хранит и показывает их по
отдельности (вкладки/секции), у каждого — своя кнопка копирования и свой набор переводов.

1. **QA / Тестировщики** — какие разделы программы изменены, затронутые фронт-роуты, изменения БД,
   обязательный регресс, зоны вероятных багов (с обоснованием по коду).
2. **Developers / Разработчики** — чья фича попала в релиз и по какой задаче (вклад по людям).
3. **Business / Бизнес** — бизнес-ценность: какие модули стали лучше, где и каким типам клиентов
   стало лучше, где выросла ценность продукта.

Полные правила содержания — в скиле. Searchify их не дублирует, только вызывает.

## 3. UX

- **Новый модуль/вью** `Release Notes` (route `/releases`), кнопка в топбаре рядом с
  `graphBtn`/`qualityBtn`/`ideasBtn`/`histBtn` (см. `src/public/index.html`).
- **Кнопка «Сгенерировать релиз»**:
  - опционально поля `from` (последний релиз — по умолчанию автодетект) и `to` (default develop HEAD);
  - по клику — асинхронный запуск (стримить прогресс стадий, как `/api/knowledge/ask/stream`:
    fetch дельты → агрегация по задачам → Jira-энрич → код-анализ → генерация 3 отчётов).
- **Результат**: три отчёта во вкладках (QA / Devs / Business) + раскрываемая секция **Baseline**
  (repo@from→to и список задач — точка отсчёта).
- **История** (переиспользовать паттерн History drawer): список сгенерированных релизов; клик открывает
  сохранённый релиз **без перегенерации** (как `openSharedAnswer`). У записи — дата, release id
  (repo@tag), кто запускал.
- **Шарабельная ссылка**: URL вида `/releases?id=<releaseId>` (и/или `?report=<reportId>`),
  открывается напрямую (по аналогии с `?answer=<id>`), `history.replaceState` на открытии.
- **Копирование (на каждый отчёт)**: кнопка «Копировать». В буфер кладётся **markdown отчёта +
  ссылка на этот отчёт в searchify** в конце (формат: `\n\n— Источник: <origin>/releases?id=…&report=<qa|dev|biz>`).
- **Перевод по запросу (на каждый отчёт)**: переключатель/кнопка языка (ru/en/fr/…). Перевод
  **не генерится заранее** — только при первом запросе на язык; затем кешируется и доступен
  мгновенно. Переключение языка не перегенерирует отчёт. Оригинал хранится как есть.

## 4. Данные (Prisma)

Новые модели (имена ориентировочные; согласовать со стилем `KnowledgeQuery`):

```
model ReleaseNote {
  id          String   @id @default(cuid())
  createdAt   DateTime @default(now())
  createdById String?               // кто запустил
  releaseId   String                // напр. "2026-06-11"
  fromRefs    Json                  // { repo: tag } точка отсчёта
  toRefs      Json                  // { repo: ref }
  baseline    Json                  // сырой список задач/коммитов (для секции Baseline)
  reports     ReleaseReport[]
}

model ReleaseReport {
  id          String   @id @default(cuid())
  releaseNoteId String
  audience    String                // 'qa' | 'dev' | 'business'
  lang        String                // язык оригинала (напр. 'ru')
  bodyMd      String                // markdown отчёта
  translations ReleaseReportTranslation[]
  releaseNote ReleaseNote @relation(fields: [releaseNoteId], references: [id])
}

model ReleaseReportTranslation {
  id        String   @id @default(cuid())
  reportId  String
  lang      String                  // целевой язык
  bodyMd    String
  createdAt DateTime @default(now())
  report    ReleaseReport @relation(fields: [reportId], references: [id])
  @@unique([reportId, lang])
}
```
`db:push` на старте создаст таблицы (миграции вручную не нужны — см. searchify-runbook).

## 5. API (Fastify, в новом `src/releases.ts`, регистрация в `src/app.ts`)

- `POST /api/releases/generate` — `{ from?, to?, repos?, lang? }` → запускает движок, создаёт `ReleaseNote`
  + 3 `ReleaseReport`. Желательно стрим-вариант `POST /api/releases/generate/stream` (SSE, как ask).
- `GET  /api/releases` — история (список, с пагинацией; только метаданные).
- `GET  /api/releases/:id` — полный релиз: baseline + 3 отчёта (+ какие переводы уже есть).
- `GET  /api/releases/report/:reportId?lang=xx` — отдать отчёт в нужном языке: если перевода нет и
  `lang` ≠ оригинал → **сгенерировать перевод on-demand**, сохранить в `ReleaseReportTranslation`,
  вернуть. Если есть — вернуть из кеша.
- Доступ: чтение — авторизованные пользователи; запуск генерации — возможно admin-only (согласовать).

## 6. Переиспользовать существующее

- **LLM**: `src/llm.ts` `runRolePrompt` (Claude CLI). Роли движка — `release_notes.qa`,
  `release_notes.dev`, `release_notes.business`, `release_notes.translate` (или один с параметром).
- **Стрим прогресса**: паттерн `/api/knowledge/ask/stream` (SSE стадий + дельта текста).
- **История + ссылка**: паттерн History drawer + `?answer=<id>` / `openSharedAnswer` /
  `history.replaceState` из `src/public/index.html`.
- **Копирование со ссылкой**: уже есть `copyAnswerLink` / `answerLinkBtn` — расширить на отчёты.
- **i18n UI-хрома**: словари `RU/EN/FR` в `index.html` (`applyTranslations`) — добавить ключи модуля.
- **Граф/код-доступ**: код лежит в `workspaces/<repo>`; граф `graph.json` для маппинга
  файлы→модули/экраны (см. searchify skill «Data Lake»).

## 7. Критерии приёмки

- [ ] Кнопка генерирует релиз; видны 3 отчёта (QA/Dev/Business) + Baseline.
- [ ] Релиз сохранён в историю; запись открывается **без перегенерации**.
- [ ] Есть прямая шарабельная ссылка на релиз и на конкретный отчёт.
- [ ] У каждого отчёта — кнопка «Копировать»; в буфере markdown **+ ссылка на отчёт в searchify**.
- [ ] Перевод отчёта генерится **только по запросу**, кешируется, переключение языка не перегенерирует.
- [ ] Оригинал и переводы одного отчёта доступны параллельно (читаемость на разных языках).
- [ ] Генерация использует движок `31-release-notes-writer` (baseline = последний prod-тег репо).
- [ ] Локальные проверки searchify пройдены и прод передеплоен (см. `skills/searchify/SKILL.md`).

## Контекст (постановка владельца, дословно сохранено)

> Релиз-нотс по всем правилам. Важен не только список коммитов с последнего релиза — это точка отсчёта,
> её нужно собрать и учитывать. Делаем для трёх ЦА: (1) тестировщики — разделы изменений, фронт-роуты,
> изменения БД, что регрессить и где вероятны ошибки (по коду); (2) разработчики — чья фича/по какой
> задаче, вклад в релиз; (3) бизнес — бизнес-ценность, какие модули/каким клиентам стало лучше.
> По кнопке генерируются отчёты, пишутся в историю со ссылкой; перевод по кнопке (on-demand);
> кнопка копирования у каждого отчёта, в скопированный текст добавляется ссылка на отчёт в searchify.
